# Security Research — Progressive Complexity Architecture ("Grow With Me") — issue #29

Scope: security/correctness risks the tier + activity-counter feature introduces, and the exact
existing error-handling / input-validation / auth-isolation patterns the implementor must mirror.
Evidence-first; every row cites file:line. Snippets ≤5 lines.

Key context discovered:

- `users.id INTEGER PRIMARY KEY AUTOINCREMENT` (starts at 1) — `036_create_auth_tables.ts:18-24`, so
  `id=0` is never a real user.
- API-key auth returns a **synthetic shared user** `{ id: 0, username: 'api' }` —
  `middleware.ts:97-102`. All API-key callers collapse to `user_id=0`.
- In `AUTH=off` / `AUTH=local` local-IP bypass, `locals.user` is `null` but the request still
  resolves (`hooks.server.ts:209-224`); the ui-preferences endpoint itself returns 401 on null user
  (`+server.ts:40-42`), so anonymous/bypass callers get safe defaults and never write.

## Acceptance Criteria additions (security)

| #   | Criterion                                                                                                                                                                                                                                |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Every tier read/write/reset query is scoped by authenticated `locals.user.id`; no query path omits `user_id` (verified per handler). No IDOR: user A cannot read/write/reset user B's tier state.                                        |
| 2   | Anonymous requests (and `AUTH=off`/`AUTH=local` bypass where `locals.user` is null) return 401 with safe defaults and perform zero DB writes — mirror `+server.ts:40-42,77-79`.                                                          |
| 3   | API-key synthetic user (`id=0`) is handled explicitly: writes are either skipped (return defaults) or guarded by `user.id > 0`; an INSERT with `user_id=0` must never surface as an unhandled FK 500.                                    |
| 4   | `tier` value is validated against allow-list enum (`beginner\|intermediate\|advanced`) at the endpoint boundary AND enforced by a DB `CHECK (tier IN (...))` constraint; invalid values → 400.                                           |
| 5   | `section_key` reuses `SECTION_KEY_PATTERN` + `SECTION_KEY_MAX_LENGTH` at the endpoint and is re-asserted in the query layer (defense in depth); invalid/oversized → 400.                                                                 |
| 6   | Activity/progression counters are clamped to a bounded non-negative integer range; non-integer, negative, or overflow inputs are rejected/clamped; DB enforces bounds via `CHECK`.                                                       |
| 7   | Counter increments and tier writes are rate-limited per `(user_id, section_key)` (mirror 8 req / 30s → 429) and increments are idempotent/bounded so user actions cannot spam unbounded DB writes.                                       |
| 8   | Concurrent tier writes use `expected_updated_at` optimistic concurrency: mismatch → 409; conditional `UPDATE ... WHERE updated_at = ?` rowcount 0 → 409; `UNIQUE constraint failed` → 409 (mirror `+server.ts:147-153,158-181,302-315`). |
| 9   | The tier context store clears cached tier state on logout / auth-identity change and on 401 (mirror `clearOnAuthChange`), so a subsequent user never inherits a prior user's tier.                                                       |
| 10  | Tier table declares `FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE` so tier + counter rows are removed on user deletion (mirror `050_...:17`).                                                                            |
| 11  | All SQL uses parameterized `?` placeholders (no string concatenation of user input) across read/write/reset paths.                                                                                                                       |
| 12  | 500 responses do not leak internal error/SQL details to clients; detailed errors are logged server-side only (tighten vs. existing `+server.ts:62,152`).                                                                                 |
| 13  | Reset endpoint gates on `locals.user`, scopes deletion by `user_id`, is rate-limited, and cannot reset other users' tiers or run anonymously.                                                                                            |

## Gaps

- GAP: No existing `tier`/`progression`/`complexity` table or endpoint in the codebase (grep of `packages/praxrr-app/src` found only unrelated upgrades/quality-profile matches) — this is a green-field feature; all patterns above are adapted from the ui-preferences family.
- GAP: Existing rate limiter (`+server.ts:37`) uses an unevicted in-memory `Map` and is per-process only — no shared/distributed rate limiting exists to reuse for high-frequency auto-progression counters. ASSUMPTION — needs validation: counter increment volume may exceed the 8/30s window designed for manual toggles.
- GAP: No existing precedent for handling the API-key `user_id=0` FK case; ui-preferences shares the same latent 500 path (never exercised because API keys rarely PATCH prefs). Policy decision required.
