## NOT Building (over-engineering risks)

- Full Setup Wizard onboarding flow: explicitly issue #12 territory; #29 is only the tier state/orchestration foundation — do NOT build a wizard here.
- Rollout of tiers to all ~15 existing feature routes: this plan is foundational; migrating every route/page to tier-awareness is scope creep — wire the primitive + 1 reference integration at most, not a full sweep.
- ML / statistical / heuristic auto-progression models: progression "suggestions" should be a simple deterministic rule (e.g. threshold on interactions), not a learned model — no analytics pipeline or scoring engine.
- Generic tier-plugin / registry / strategy abstraction: rule-of-three not met (tiers are a fixed known set of exactly 3). Model as a fixed `['beginner','intermediate','advanced'] as const` union + simple map mirroring `UI_PREFERENCE_MODES`, NOT a pluggable registry — speculative generality.
- New tier persistence mechanism: existing per-user per-section DB table + optimistic-concurrency + debounced client store already covers "per-user per-section state" — do NOT invent a parallel store, cache layer, or cross-device sync beyond the existing DB row.
- Cross-device tier sync / realtime push beyond DB: no websockets/broadcast; persistence is the existing request/response + hydrate-on-load pattern only.
- Analytics dashboards / usage telemetry for progression: out of scope; no new metrics tables or reporting UI.
- Full `$ui` design-system tier token system: no existing tier tokens found (grep for tier/complexity → none in lib). Reuse the `AdvancedSection`/`DisclosureSection` primitives; do NOT author a new theming/token subsystem in the foundational plan.
- Reset feature as a bespoke bulk endpoint: "reset" maps to writing tier back to the default via the existing upsert/PATCH path (mirror `setUiPreference` to default) — do NOT design a new reset RPC unless bulk-per-user reset is explicitly required.
- Renaming/replacing the existing 2-mode `basic/advanced` system: tiers layer ON TOP; the existing `UiPreferenceMode`, `user_interface_preferences` table, store, route, and `SECTION_KEY_PATTERN` must be reused, not duplicated or reinvented.

## Gaps

- GAP: No existing tier/complexity/beginner/intermediate code anywhere in `packages/praxrr-app/src` (grep → only unrelated score-simulator matches). This is greenfield layered on the disclosure primitive.
- GAP: No existing `$ui` design-system "tier" tokens or theming; only `AdvancedSection`/`DisclosureSection` primitives exist. Issue #29's "$ui tier integration" has no prior art to mirror beyond these components.
- AMBIGUITY — needs validation: query-module file naming. Disclosure uses snake_case (`user_interface_preferences.ts`); ~all other 34 query modules use camelCase. Implementor must pick one; mirroring the disclosure feature = snake_case, mirroring the dominant convention = camelCase.
- GAP: Migration `version` for a new date-based migration is the integer `YYYYMMDD`; no example collides on same-day multiple migrations — if two land same day, disambiguation scheme is unspecified in-repo.
