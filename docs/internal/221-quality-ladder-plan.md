## Implementation plan — Issue #221: Quality Goals · Quality Ladder · Enforce Resolution Ceiling (revised)

### 0. Scope corrections applied to the design (read before starting)

The design's §4.6/§5 apply model ("merge scoring+ladder into ONE `writeOperation` with a combined `desiredState { custom_format_scores, ordered_items }`") is **superseded** by the verified simplification:

- Apply builds `operations: [...scoringOps, ladderBatchedOp]` where each entry is a single-family op literal (`{ sql, metadata, desiredState }`), passed to **one** `writeOperationsFromSql(...)` call with `runValueGuardGate: true`.
- `writeOperationsFromSql` (writer.ts:741) already validates ALL sql up front (`cache.validateSql`), runs `runValueGuardGate` over EVERY op (each evaluated by its own single-family `desiredState` — scoring op → `custom_format_scores` rule; ladder op → `ordered_items` full-list rule), then persists all ops + recompiles ONCE. A guard mismatch on ANY op → `{ success:false }` and NOTHING persists.
- **DO NOT** modify the core guard gate. **DO NOT** merge scoring+ladder into a single op. Each op keeps its own single-family desiredState.
- `MigrationSqlOperation`/`WriteSqlOperationsOptions` are module-private; build the operations array as **inline object literals** and `import { writeOperationsFromSql } from '$pcd/ops/writer.ts'` directly (not via the `$pcd` barrel).
- Registry-cache parity: `runValueGuardGate` re-executes ops against `getCache(databaseId)`. `buildGoalPlan` and `persistGoalApply` both read the same `pcdManager.getCache(request.databaseId)` cache.

**Design symbol/path bugs to ignore:** the apply handler is `handleGoalApplyRequest` (apply/+server.ts:32), not `_handleGoalApplyRequest`. `describeChange` (impact helpers) is not on the goals path and is unchanged.

### 0.0 Findings resolution (baked into the batches below)

- **F1 (blocker):** `buildGoalApplyOps` now takes an **options object including `databaseId: number`** and forwards it to `buildScoringOps`/`buildQualityLadderOps`; `persistGoalApply` passes `options.databaseId` through (the apply handler already holds `goalRequest.databaseId`). Fixes `TS2304`.
- **F2 (blocker):** `toProfileEdit` sets `ladderInput: plan.ladderInput ?? undefined` so `null` never flows into the `?: UpdateQualitiesInput` (undefined-only) field. Fixes `TS2322` at the ProfileEdit boundary.
- **F3 (major):** `buildGoalApplyOps` uses an **explicit return type** whose element `metadata` is `OperationMetadata` (import type from `$pcd/ops/writer.ts`), so `operation: 'update'` is contextually typed (not widened to `string`) and the array assigns to the private `MigrationSqlOperation[]`.
- **F4 (major):** apply's echoed `configDiff` is computed through the **same ephemeral sandbox path as preview** via a shared `computeGoalConfigDiff` helper (new in b6), captured BEFORE `persistGoalApply` — never by re-diffing the mutated live cache against itself. This makes apply `configDiff === preview configDiff` and satisfies AC3. Consequently **b5 (apply) now dependsOn b6 (sandbox)**.
- **F5 (minor):** every `buildQualityLadderOps` consumer (b2 `updateQualities`, b6 `withSandboxCache`) narrows with `if ('error' in built) { ... }` first, then guards `if (built.batched === null)` before touching `built.batched.queries`.
