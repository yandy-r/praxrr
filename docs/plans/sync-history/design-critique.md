# Design Critique: Sync History (#17)

**Verdict:** ready-with-fixes

## Blocking issues (resolved in design.md §0)

- Detail-capture (section 2.2/3) is unsound as written. The doc claims full before/after `changes` are 'already held in-process' in each syncer's `sync()` and capturable via a 'thin return extension with no extra Arr GETs, reusing the same diffSingletonEntity/diffEntityCollection calls generatePreview uses.' The code disproves this: (a) QP `sync()` (qualityProfiles/syncer.ts:246-335) never calls diffEntityCollection — only generatePreview() does (lines 397-420) — and the diffable desired collections (desiredCustomFormats/desiredProfiles) are assembled ONLY in generatePreview (lines 364-395); `sync()` builds+writes each payload inline (877/884). (b) The cited 'current' CF fetch at line 277 happens AFTER CFs are written at 266-274, so it is a post-write snapshot, invalid as a before-baseline. (c) Delay `sync()` fetches current only for Lidarr (delayProfiles/syncer.ts:282/318-320); radarr/sonarr fetch no current at all. Consequently full-diff capture requires each syncer to assemble desired collections AND take pre-write current snapshots — a substantive refactor with added Arr round-trips, not a thin return extension. Fix: either (1) explicitly scope Phase 1 to counts + section_results and defer full field-level diffs to a follow-up, or (2) rewrite section 2.2/3 to specify, per syncer and per arr_type, where a pre-write current snapshot and desired-collection assembly must be added (accepting the extra GETs).
- Status enum / DDL omit the handler's `cancelled` terminal exit. The doc asserts the recorder covers 'every terminal exit' and lists 'disabled/missing :300', but arrSync.ts:300 returns status 'cancelled' (and JobRunStatus includes 'cancelled', queueTypes.ts:21). The proposed sync_history.status CHECK allows only ('success','partial','failed','skipped'), and record.ts is specified to never throw — so a disabled-instance run would either silently violate the CHECK and be dropped, or be unrecorded. Decide and document: map cancelled->skipped, add 'cancelled' to the enum, or explicitly do-not-record disabled-instance runs.

## Improvements

- The 4-value status mapping keys 'partial' on itemsSynced (>0), but arrSyncHandler does not currently accumulate itemsSynced — it tracks only `failures`, `ranSections`, and a `results: string[]` (arrSync.ts:370-372). Call out that the recording refactor must sum itemsSynced from the retained per-section SyncResult objects; otherwise the partial/failed split is uncomputable.
- sync/types.ts already declares BaseSyncer.sync(): Promise<SyncResult> (types.ts:116) and syncers import SyncResult from ../base.ts (qualityProfiles/syncer.ts:18). Clarify that extending SyncResult itself (types.ts) is sufficient for the interface — the doc redundantly lists both 'extend SyncResult' and 'extend the four sync() overrides', which risks a double-typed contract.
- trigger_event plumbing is described as an in-scope 'small enhancement' threading the event through triggerSyncs/processor, but the recording path and the fallback (ship always-NULL) both work without it. Recommend hard-deferring trigger_event to keep this PR bounded, given the diff-capture refactor already expands scope.
- The design does not address whether recordSyncHistory should fire from executeSyncJob's synthetic job.id=0 path in a way that populates job_id meaningfully — job_id will be 0 for all manual runs (arrSync.ts:105). Note this so the UI/export don't treat 0 as a real queue id.
- Consider an explicit note that MediaManagement's flattened single SyncResult (deferred per the doc) means its history rows will have section_results but likely no `changes` — the detail view's degrade-to-section_results path must be built and tested, not just mentioned.
- Scope realism (e): even setting aside the diff-capture refactor, this PR spans a migration + 2 query modules + recorder/responses/types + 4 syncer edits + cleanup job + 4 API routes + OpenAPI + 4 UI files + notifications in 2 files + 5 test files + shared registrations. This is large for one PR; recommend splitting (Phase 1: table + recorder + counts + list/detail API/UI + retention; Phase 2: full diffs + notifications + export).
