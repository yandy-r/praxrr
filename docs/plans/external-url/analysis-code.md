# Code Analysis: external-url
## Executive Summary
The `external_url` work touches the Arr instance domain at the database, loader, action, and UI layers. The current stack treats `url` as the single source of truth for both backend clients (library/library test/PCD sync jobs) and browser "open" links, while the feature spec wants a user-configurable override that only affects UI entry points. Any solution must therefore keep `url` as the canonical communication endpoint (used by `arr/test`, `api/v1/arr/library`, and `createArrClient`), while plumbing an `external_url || url` resolver through every surface that renders or opens instance links (forms, list/table/card views, library action bar, library rows). Database migrations, query types, and hidden forms must absorb the new column without changing the existing Arr-type wiring.
## Existing Code Structure
`/src/lib/server/db/schema.sql` defines `arr_instances` with `id`, `name`, `type`, `url`, `api_key`, `tags`, `enabled`, and timestamps; all Arr-specific tables (sync configs, namespaces, rename settings, etc.) cascade from `arr_instances`. The migration runner in `src/lib/server/db/migrations.ts` statically imports every migration module, so adding `external_url` requires a new migration file plus an import entry there. CRUD logic lives in `src/lib/server/db/queries/arrInstances.ts`: a central `arrInstancesQueries` object exposes strongly typed `ArrInstance`, `CreateArrInstanceInput`, `UpdateArrInstanceInput`, plus helpers that JSON-encode `tags`, enforce enabled flags, and dynamically build `UPDATE` statements.
SvelteKit routes consume `arrInstancesQueries`: `/routes/arr/new/+page.server.ts` enforces PRG, validates required `name`, `type`, `url`, `api_key`, ensures unique names/API keys, optionally applies default delay profiles, and redirects to the instance settings page upon creation. `/routes/arr/[id]/settings/+page.server.ts` fetches the row, validates required fields, and delegates to `arrInstancesQueries.update`, while delete flows trigger `cleanupJobsForArrInstance`. `[id]/+layout.server.ts` loads the instance once per route subtree and surfaces it as `data.instance` for the library and nested pages.
The UI uses `InstanceForm.svelte` for the shared add/edit experience, with `initEdit`/`dirty` stores tracking `name`, `type`, `url`, `api_key`, `enabled`, and `tags`. The form hides real submission via `<form id="save-form" use:enhance>`, and the hidden inputs mirror the tracked fields so that `handleSave`/`handleConnection` can test the connection (`/arr/test` uses `createArrClient(type, url, apiKey)` with canonical `url`). `/routes/arr/[id]/library/+page.svelte` receives `data.instance`, derives `instanceUrl = instance?.url?.replace(/\/$/, '')`, and feeds that base to row actions and to `handleOpen` via `LibraryActionBar`. Table and card views under `/routes/arr/views/` consume `instances: ArrInstance[]` directly, displaying `instance.url` in badges and using it for `window.open` on external-link buttons. The library action barâ€™s `onOpen` handler simply opens `instanceUrl` in a new tab, and the Open-in links in the table defaults derive from `baseUrl = instanceUrl`.
## Implementation Patterns
`arrInstancesQueries` is the single SQL source of truth and is imported anywhere an instance row is needed (actions, loaders, UI stores). Create and update entries live entirely within this module; actions rely on the typed DTOs and never craft SQL inline. SvelteKit actions uniformly `fail(400, {...})` on validation errors and `redirect(303, ...)` on success, while background jobs like delay-profile application run after the database insert to avoid blocking. Forms use `$alerts` and `enhance` to manage optimistic UI state and rely on `dirty` tracking to enable the Save button only when required fields (`name`, `url`, `api_key`) are present. The `(mode === 'edit' ? '?/update' : undefined)` submission target determines which action is hit, and hidden inputs synchronize the reactive store with the actual payload. For UI navigation, `[id]/+layout.server.ts` loads the instance exactly once and recycles it across nested layouts, so any new field must flow through that loader before components derive fallback URLs. Client-side components trim trailing slashes (`instance?.url?.replace(/\/$/, '')`) before building URLs, and the action bar/table/card components open those strings in `_blank` windows without any additional logic.
## Integration Points
- **Database**: `arr_instances` currently lacks `external_url`; the column must be added via a migration and tracked in the `schema.sql` documentation. Foreign-key tables (sync configs, delay profiles, rename settings, namespaces) already cascade from `arr_instances`, so no schema changes upstream are needed aside from referencing the new column.
- **Queries**: `ArrInstance`, `CreateArrInstanceInput`, `UpdateArrInstanceInput`, and the `arrInstancesQueries.create/update/get*` helpers must expose `external_url`, ensure the INSERT/UPDATE statements include it (and allow a null/empty override), and surface it to callers.
- **Forms**: `InstanceForm.svelte` initializes the `dirty` store with the tracked fields; `mode === 'edit'` prepopulates `url`, so it must also prepopulate `external_url` (but keep `apiKey` blank). Hidden `<input>` elements in the save form must mirror the new field so server-side actions receive it. `handleSave` should not require the optional field, but the store should update when the user types it.
- **Actions**: `/routes/arr/new/+page.server.ts` and `/routes/arr/[id]/settings/+page.server.ts` parse `external_url` (probably trimming and setting `null` when empty) alongside the existing `name`, `url`, `api_key`, and pass it to `arrInstancesQueries.create/update`. Validation should stay focused on required fields (`name`, `type`, `url`, `api_key`) while allowing `external_url` to be empty.
- **Layout**: `[id]/+layout.server.ts` just returns `instance`, so as long as `arrInstancesQueries.getById` returns the new column, child routes and components (library, settings) automatically receive it.
- **Library UI**: `instanceUrl` and `baseUrl` currently derive solely from `instance.url`. `handleOpen`, `LibraryActionBar`, and the `radarr/sonarr/lidarr` row action anchors all need to resolve a single fallback (`instance.external_url || instance.url`, trimmed of trailing slashes) for browser navigation. `LibraryActionBar` does not know about `instance`, so the base URL must be computed upstream and passed down (e.g., `onOpen` could close over the resolved base).
- **List Views**: `TableView.svelte` and `CardView.svelte` render badges for `instance.url` and open `window.open(row.url, '_blank')`. These action buttons must instead use the resolver so the cards/list share the same fallback and show whichever URL is used for navigation.
- **Server Clients**: `arr/test/+server.ts` (connection testing) and `/routes/api/v1/arr/library/+server.ts` (library fetching) instantiate Arr clients using `instance.url`, and the factory in `src/lib/server/utils/arr/factory.ts` switches on type. This reinforces that backend traffic must continue to rely on the canonical `url`, while `external_url` only affects UI links.
## Code Conventions
- SQL models adopt snake_case for column names and store JSON arrays as `TEXT` (see `tags` handling). `arrInstancesQueries` returns rows with `created_at`/`updated_at`, and `update` statements append `updated_at = CURRENT_TIMESTAMP` automatically.
- TypeScript DTOs for create/update operations translate camelCase input (e.g., `apiKey`) to snake_case columns (`api_key`). All optional fields are typed to allow omission, and update helpers only append parameters when values are defined.
- Forms follow the SvelteKit `enhance` + hidden `<form>` pattern with manual `requestSubmit`, and `dirty`/`alertStore` track form validity and server responses. API responses are handled via `alertStore.add('success'/'error', ...)` depending on redirect vs failure.
- Validation is defensive: actions log warnings via `$logger` before returning `fail(400)`, ensuring duplicate names/API keys are rejected early.
- UI code trims trailing slashes (via `.replace(/\/$/, '')`) before using URLs and uses `window.open(..., '_blank', 'noopener,noreferrer')` consistently for security.
## Dependencies and Services
- `arrInstancesQueries` depends on the database module (`src/lib/server/db.ts`) and is used by route actions, layouts, and server APIs.
- Connection checks and delay-profile updates rely on `$arr/factory.ts` + `$arr/clients/{radarr,sonarr,lidarr}.ts`, so the feature must not change which URL is passed to those clients.
- `/routes/api/v1/arr/library/+server.ts` uses `RadarrClient`, `SonarrClient`, and `LidarrClient` directly, alongside `pcdManager`, `qualityProfileQueries`, and the caching layer (`$cache/cache.ts`). The cache key is scoped only by `instanceId`, so UI overrides of `external_url` do not affect cached data.
- The form/UI layer uses shared component libraries (`$ui/form`, `$ui/actions`, `$ui/table`), capability metadata (`$shared/arr/capabilities.ts`), and stores like `$stores/libraryCache` and `$stores/search`, so any new prop (e.g., resolved open URL) must flow cleanly through these abstractions without breaking existing props.
## Gotchas and Warnings
1. `external_url` must not leak into any backend Arr client initializationâ€”they must keep using `instance.url` to preserve connectivity. The client factory, `/arr/test`, and `/api/v1/arr/library` are the canonical spots to confirm this.
2. The InstanceForm intentionally does not pre-populate the API key for edit mode (`apiKey` stays `''`), so adding `external_url` should mirror other non-sensitive fields (prepopulate for edit, start empty for create) but stay optional.
3. Hidden forms (`save-form`) currently track six inputs; the new field needs to participate without deviating from the `enhance` hook (which toggles `saving` and listens for `form.success`). Forgetting to add the `<input name="external_url" ...>` would mean the server never sees the override.
4. The update query builder only adds columns when the value is defined; make sure empty strings map to `null` or are skipped so that clearing the external URL removes it instead of storing an empty string.
5. Library rows and the toolbar rely on `instanceUrl` for multiple link builders (movie/series/artist). The fallback zipper (`external_url || url`) should be computed once (with slash-trimming) to avoid inconsistencies between the toolbar button and the row-specific external-link buttons.
## Task-Specific Guidance
1. **Schema/Migrations**: Add `external_url TEXT` (nullable) to `arr_instances` via a new migration under `src/lib/server/db/migrations/` and include the migration import in `src/lib/server/db/migrations.ts`. Update `schema.sql` documentation to show the new column.
2. **Queries**: Extend `ArrInstance`, `CreateArrInstanceInput`, and `UpdateArrInstanceInput` with `externalUrl?: string | null`. Update `create` to insert `external_url` (coerce empty strings to `null`) and `update` to add `external_url = ?` when provided. Ensure `arrInstancesQueries.get*` automatically exposes the column so the loader can read it.
3. **InstanceForm**: Track `external_url` in `dirty` state (prepopulate when editing, start empty when creating), expose a new visible input (with optional description/validation), and add a hidden `<input name="external_url" ... />` in the save form. Keep `canSubmit` logic unchanged (the field is optional) but ensure `update('externalUrl', ...)` and `initEdit` reflect the new value.
4. **Actions**: `/routes/arr/new/+page.server.ts` should `const externalUrl = formData.get('external_url')?.toString().trim();` (set to `undefined`/`null` when empty) and pass it to `arrInstancesQueries.create`. `/routes/arr/[id]/settings/+page.server.ts` should do the same before calling `update`. Consider logging when an external URL is provided/cleared for telemetry if desired.
5. **UI Resolver**: Let `[id]/library` compute a single `resolvedInstanceUrl = (instance?.external_url || instance?.url)?.replace(/\/$/, '')` and feed that into both `handleOpen` and the `LibraryActionBar` (pass as a prop or close over it) so every Open-in entry point uses the same base.
6. **List Views**: Update `TableView.svelte` and `CardView.svelte` to resolve the fallback URL per row (maybe via a helper like `const resolvedUrl = instance.external_url?.trim() || instance.url;`). Display the canonical `url` in badges (if desired) but open the fallback URL when the external-link button is clicked.
7. **Preserve Backend Flow**: Leave `/routes/arr/test/+server.ts` and `/routes/api/v1/arr/library/+server.ts` untouched with respect to which URL they send to Arr clientsâ€”they must keep using the canonical `url` column for communication.
8. **Documentation/Spec**: Revisit `docs/plans/external-url/feature-spec.md`, `research-technical.md`, and `research-ux.md` before implementation to align validation messaging, fallback semantics, and rollout notes with the latest requirements.
