# Pattern Research: external-url

## Architectural Patterns

- **Incremental schema evolution**: `docs/plans/external-url/feature-spec.md` and the accompanying technical notes call for a single `ALTER TABLE` migration (`packages/praxrr-app/src/lib/server/db/migrations/0XX_add_arr_instances_external_url.ts`) that adds a nullable `external_url` column on `arr_instances`. The pattern matches the expand/contract guidance (add column first, keep it nullable, deploy code that reads both old and new structures) so existing deployments stay live while the new URL flows behind a feature flagged field.
- **Query-layer coverage**: `packages/praxrr-app/src/lib/server/db/queries/arrInstances.ts` already centralizes the canonical `ArrInstance` shape, create/update input types, JSON-serializes `tags`, and normalizes `enabled`. Extending this module to expose `external_url` (trimmed input, `null` for empty strings, updated `ArrInstance` interface, and helper guards in `create`/`update`) follows the repo’s single-source-of-truth pattern for Arr metadata.
- **SvelteKit action funnel**: `packages/praxrr-app/src/routes/arr/new/+page.server.ts` and `packages/praxrr-app/src/routes/arr/[id]/settings/+page.server.ts` share the create/update flow: parse trimmed `FormData`, validate required fields, short-circuit duplicates via `arrInstancesQueries.nameExists`/`apiKeyExists`, wrap the write in a `try/catch` for logging, `return fail(...)` on validation errors, and `redirect(303, …)` on success. Adding `external_url` reuses this funnel, treating the field as optional (validate when provided, normalize to `null`, store metadata for logging) so the majority of the action structure stays unchanged.
- **UI link-resolution helper**: `docs/plans/external-url/research-technical.md` records how every `Open in` surface (`packages/praxrr-app/src/routes/arr/[id]/library/+page.svelte`, `components/LibraryActionBar.svelte`, Card/Table views) should derive the link as `instance.external_url?.trim() || instance.url`. That shared resolver keeps browser navigation decoupled from backend traffic, keeping `arr/test` and syncing clients pointed at `instance.url` while UI links prefer the presentation-only `external_url`.

## Code Conventions

- **File organization & naming**: Server queries live under `packages/praxrr-app/src/lib/server/db/queries/*` and export descriptive helper objects (e.g., `arrInstancesQueries`). Routes place actions in `+page.server.ts` with `export const actions` typed as `Actions`, while UI components live in `/components` directories and reuse shared stores (`$lib/client/stores/dirty.ts`, `$alerts/store`). Maintaining these conventions means the new `external_url` plumbing lives in the existing `arrInstancesQueries` module and the `arr` routes/components instead of introducing new scattered helpers.
- **Type safety & normalized inputs**: `arrInstancesQueries` declares `ArrInstance`, `CreateArrInstanceInput`, and `UpdateArrInstanceInput` interfaces and avoids `any`. Inputs like `tags` are JSON-serialized inside the query helper, and booleans like `enabled` map to integers. Adding `external_url` should follow the same pattern: expand the interfaces, treat the field as `string | null`, trim/normalize inside the action layer, and persist through the query helper without introducing `any`.
- **Progressive-enhancement forms**: `packages/praxrr-app/src/routes/arr/components/InstanceForm.svelte` uses `use:enhance`, derived `$current` dirty-store values, and explicit `alertStore` calls for connection-test feedback. The component imports shared `ARR_APP_OPTIONS`, `FormInput`, `DropdownSelect`, and `TagInput` helpers. Extending this form with an optional `External URL` field means adding another controlled input that wires into the dirty store, stays hidden from the server when empty, and optionally surfaces its value via a `<input type="hidden" name="external_url" ...>` so the enhanced form keeps submitting the data.

## Error Handling

- **Server validation via `fail`**: The actions in `packages/praxrr-app/src/routes/arr/new/+page.server.ts` and `packages/praxrr-app/src/routes/arr/[id]/settings/+page.server.ts` validate required fields, run duplicate checks, log warnings, and `return fail(400, { error, values })` when inputs are invalid. This keeps the entered values inside the `form` store so the UI can re-render with the previous inputs, and the PRG-style `redirect(303, …)` success path avoids double submits.
- **Selective `try/catch` usage**: The `arr/[id]/settings` update wraps only the query update in a `try/catch` to log unexpected failures (e.g., DB write errors) and respond with `fail(500, { error: 'Failed to update instance' })`. All expected validation errors are handled outside the catch block so the action treats predictable problems as data errors instead of crashes.
- **Logging context**: Each failure logs `source`, `meta`, and sanitized payload (e.g., `hasApiKey` boolean, instance `type`) so operations can be traced without exposing secrets. New `external_url` writes should log whether the field was set/cleared while keeping the canonical `url` unchanged.

## Testing Approach

- **Unit tests call actions directly**: `packages/praxrr-app/src/tests/base/lidarrOnboarding.test.ts` imports `actions` from `packages/praxrr-app/src/routes/arr/new/+page.server.ts`, patches `arrInstancesQueries` helpers, builds `FormData` requests, and asserts that invalid inputs hit `fail(400, …)` while valid submissions trigger a `303` redirect. This mirrors the general recommendation to treat form actions as ordinary functions for tests (mocking dependencies as needed) so validation logic is exercised without spinning up a server.
- **E2E coverage for settings/library flows**: `packages/praxrr-app/src/tests/e2e/specs/2.40-lidarr-core-flow.spec.ts` navigates `/arr/new`, adds instances, and verifies arr library behavior. Adding `external_url` should extend such specs to confirm the `LibraryActionBar`/`Open in` flows now prefer the new field when present and fall back to `url` when it is empty.
- **Job-level readability**: `packages/praxrr-app/src/tests/jobs/lidarrSync.test.ts` already checks instance enablement messages, so additional tests can ensure syncers still read `instance.url` even when `external_url` exists.

## Patterns to Follow

- **Browser-only `external_url`**: Keep backend clients (`arr/test`, sync jobs, `createArrClient`) wired to `arr_instances.url`. Use `external_url` solely for UI link building and persist it alongside the existing metadata so everything stays co-located and query-friendly.
- **Normalize empties to null**: Trim the field, write `null` when the user clears it, and ensure `arrInstancesQueries` sees `null` instead of an empty string, following the normalization note in the feature spec (`external_url` cleared reverts links to `url`).
- **Shared resolver/helper**: Put the fallback expression (`external_url?.trim() || url`) in one helper or maintain it within shared state so every `Open in` button reuses the same logic—this keeps surfaces (toolbar buttons, card/table rows) consistent with the documented behavior.
- **Mirror existing validation paths**: Integrate the new field in `InstanceForm`, `arr/new`, `arr/[id]/settings`, and any other edit paths concurrently. Use the same `validate required fields → duplicate checks → try/catch logging → redirect` pattern already in place, so reviewers can verify the change without chasing multiple styles.
