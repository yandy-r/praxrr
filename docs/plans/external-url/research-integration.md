# Integration Research: external-url

## API Endpoints

### Existing Related Endpoints

- `POST /arr/new` (`packages/praxrr-app/src/routes/arr/new/+page.server.ts`) backs the InstanceForm creation flow, validates `name/type/url/api_key`, writes a row via `arrInstancesQueries.create`, and briefly spins up `createArrClient` to apply the default delay profile for Radarr/Sonarr instances before redirecting to `/arr/{id}/settings`.
- `POST /arr/[id]/settings` (`packages/praxrr-app/src/routes/arr/[id]/settings/+page.server.ts`) handles edits and deletes. Both actions re-read the row via `arrInstancesQueries.getById`, update the writable fields, clear cached jobs (`cleanupJobsForArrInstance`) on delete, and log the `instance.url` metadata that also drives the “Open in” links.
- `POST /arr/delete`/`GET /arr` (`packages/praxrr-app/src/routes/arr/+page.server.ts`) renders the landing list of instances (`arrInstancesQueries.getAll`) and exposes a server action that reuses the same query module plus `cleanupJobsForArrInstance` before redirecting.
- `POST /arr/test` (`packages/praxrr-app/src/routes/arr/test/+server.ts`) performs a fast connection check by calling `createArrClient` with the supplied `url`/`apiKey` so the UI can validate credentials before writing to the database.
- `GET /api/v1/arr/library` + `DELETE /api/v1/arr/library` (`packages/praxrr-app/src/routes/api/v1/arr/library/+server.ts`) resolve `instanceId` via `arrInstancesQueries.getById`, create `RadarrClient`/`SonarrClient`/`LidarrClient`, fetch the library with filtering/sorting/pagination, cache the result, and always call the Arr API at `instance.url`. The delete handler simply invalidates `getArrLibraryCachePrefix` for the instance.
- `GET /api/v1/arr/library/episodes` (`packages/praxrr-app/src/routes/api/v1/arr/library/episodes/+server.ts`) validates that the instance is Sonarr and builds a `SonarrClient(instance.url, instance.api_key)` before fetching season/episode details for the expanded rows on the library page.
- `GET /api/v1/arr/releases` (`packages/praxrr-app/src/routes/api/v1/arr/releases/+server.ts`) uses the same `instance.url` to spawn Radarr/Sonarr/Lidarr clients and run grouped release searches per media item.
- `POST /api/v1/arr/cleanup` (`packages/praxrr-app/src/routes/api/v1/arr/cleanup/+server.ts`) tests or executes cleanup of namespace-suffixed entities by sending the request through `createArrClient(instance.url, ...)`.
- The `/arr/{id}/logs` load action (`packages/praxrr-app/src/routes/arr/[id]/logs/+page.server.ts`) also instantiates an Arr client with `instance.url`/`api_key` to page through the remote log stream, so any change that touches `instance.url` can surface in that view.
- `packages/praxrr-app/src/routes/arr/[id]/library/+page.server.ts` exposes a `delete` action that mirrors the landing-page delete plus cache invalidation. The client-side script in `packages/praxrr-app/src/routes/arr/[id]/library/+page.svelte` reads the layout-provided `instance` object and opens `instance.url` in a new tab inside the `handleOpen` helper, so the existing API surface pairs API calls with an `instance.url`-driven “Open in app” UX.

### Route Organization

- `/arr` is the root listing page whose `load` handler pipes `arrInstancesQueries.getAll()` to the `CardView`/`TableView` components (`packages/praxrr-app/src/routes/arr/views/CardView.svelte` and `packages/praxrr-app/src/routes/arr/views/TableView.svelte`). Those cards show `instance.url`, render the `ExternalLink` button that calls `window.open(instance.url, '_blank')`, and trigger `arr/[id]` navigation for each instance card.
- Creation and editing share `packages/praxrr-app/src/routes/arr/components/InstanceForm.svelte`. The form currently tracks `name`, `type`, `url`, `api_key`, `enabled`, and `tags` (with JSON serialization). The component uses the `enhance` helper to submit through `/arr/new` or `/arr/[id]/settings` and also posts to `/arr/test` for connection verification before saving.
- The `[id]` subtree is anchored by `packages/praxrr-app/src/routes/arr/[id]/+layout.server.ts`, which fetches a single `instance = arrInstancesQueries.getById(id)` and exposes it to every child route (`library`, `logs`, `rename`, `sync`, `upgrades`, etc.). Because this layout runs after every `?/update` action, any new `external_url` value would just appear in the `instance` object without extra wiring.
- The `/arr/[id]/library` page (`+page.svelte`) reuses `LibraryActionBar` (`packages/praxrr-app/src/routes/arr/[id]/library/components/LibraryActionBar.svelte`) so its toolbar “Open in …” button delegates to the `handleOpen` helper that currently invokes `window.open(instance.url, ...)`. Whenever the action is triggered, the query builder inside `/api/v1/arr/library` uses the same `instance.id`/`instance.url` so the data and the UI link share the source row.
- The `[id]/rename`, `[id]/sync`, and `[id]/upgrades` pages also rely on the layout-provided `instance` to read `instance.name`, `instance.type`, and `instance.url`, even though they don’t open the external UI directly. Any new `external_url` field added to `instance` will therefore be available to every route without additional fetches.

## Database

### Relevant Tables

- `arr_instances` (`packages/praxrr-app/src/lib/server/db/schema.sql:19-42`, migration `001_create_arr_instances.ts`) stores every Arr connection (Radarr, Sonarr, Lidarr, Readarr, Prowlarr). It is the only canonical location for the base URL, API key, friendly name, enabled flag, and tag metadata used by the UI, jobs, and sync tables.
- Sync configuration tables reference `arr_instances`: `arr_sync_delay_profiles_config` (`schema.sql:433-452`) and `arr_sync_media_management` (`schema.sql:455-479`) both use `instance_id` as their primary key plus foreign keys to `database_instances`, so any sync run/setting is scoped to the `arr_instances` row that supplies the credentials.
- `arr_database_namespaces` (`schema.sql:482-495`) indexes `(instance_id,database_id)` namespaces for sync suffixes, `arr_rename_settings` (`migrations/024_create_arr_rename_settings.ts`) keeps one rename config per instance, and similarly `arr_upgrade_runs`, `arr_sync_*` tables, and job tables all cascade on `arr_instances(id)` so deleting/configuring an Arr instance ripples through those dependents.

### Schema Details

- `arr_instances` columns (from `schema.sql:24-41`): `id` (PK), `name` (unique user label), `type` (`radarr|sonarr|readarr|lidarr|prowlarr`), `url` (base API/path used by every Arr client and the current “Open in” links), `api_key`, optional `tags` JSON, `enabled` flag (default `1`), and `created_at`/`updated_at` timestamps. The forthcoming `external_url TEXT` column (per `docs/plans/external-url/research-technical.md`) will sit beside `url`, remain nullable for backward compatibility, and propagate through `arrInstancesQueries.getAll`/`getById`.
- `arr_sync_media_management` (`schema.sql:460-479`) stores the three configured database+config pairs (`naming`, `quality_definitions`, `media_settings`), sync trigger/cron metadata, and the `instance_id` foreign key to `arr_instances`. Any change to `arr_instances.url`/`external_url` should not affect these sync tables, but the `instance_id` tie means new UI presets (like an external link override) automatically follow the same sync configuration because the job runner fetches the entire row before creating clients.
- `arr_sync_delay_profiles_config` (`schema.sql:439-452`) shares the same pattern, while `arr_database_namespaces` (`schema.sql:487-495`) and `arr_rename_settings` (`migrations/024_create_arr_rename_settings.ts`) declare `FOREIGN KEY (instance_id)`/`arr_instance_id` with `ON DELETE CASCADE`, so cleanup actions triggered by deleting an instance rely on `arr_instances` being the “source of truth.”

## External Services

- All third-party Arr APIs (Radarr, Sonarr, Lidarr, plus Chaptarr when present) are accessed through `createArrClient` in `packages/praxrr-app/src/lib/server/utils/arr/factory.ts`, which selects the right client implementation from `packages/praxrr-app/src/lib/server/utils/arr/clients/{radarr,sonarr,lidarr,chaptarr}.ts`. Each client is constructed with `instance.url`, so keeping that field accurate is critical for every sync/release/log/cleanup call.
- The library endpoint, release search, episode details, cleanup, log viewer, rename/upgrade jobs, and the `/arr/test` connection probe all instantiate an Arr client with the same `instance.url`/`api_key` pairing. None of those server-side flows should switch to `external_url` because the backend must keep talking to the internal API endpoint (`docs/plans/external-url/research-technical.md`).

## Internal Services

- `packages/praxrr-app/src/lib/server/db/queries/arrInstances.ts` centralizes CRUD, existence checks (`nameExists`, `apiKeyExists`), and JSON serialization for `tags`. It will need to extend the exported `ArrInstance`, `CreateArrInstanceInput`, and `UpdateArrInstanceInput` types to carry the new `external_url` column alongside `url`.
- UI flows (`packages/praxrr-app/src/routes/arr/new`, `packages/praxrr-app/src/routes/arr/[id]/settings`, `packages/praxrr-app/src/routes/arr/+page.server.ts`) rely on that query module for every write/read; the library route loads `instance` via the layout (`packages/praxrr-app/src/routes/arr/[id]/+layout.server.ts`), so each child page automatically sees the added field once `arrInstancesQueries.getById` returns it.
- Library UI components (`packages/praxrr-app/src/routes/arr/[id]/library/+page.svelte` and `packages/praxrr-app/src/routes/arr/[id]/library/components/LibraryActionBar.svelte`) currently call `handleOpen` (which invokes `window.open(instance.url, '_blank')`) to expose the “Open in …” action. The `CardView` landing-page component (`packages/praxrr-app/src/routes/arr/views/CardView.svelte`) duplicates the same behavior in the instance list, so both should read `instance.external_url ?? instance.url` once the database exposes it.
- Server-side sync job handlers (`packages/praxrr-app/src/lib/server/jobs/handlers/arrSync.ts`, `packages/praxrr-app/src/lib/server/sync/processor.ts`) wrap `createArrClient(instance.type, instance.url, instance.api_key)` before running sync sections. Rename/upgrade processors (`packages/praxrr-app/src/lib/server/rename/processor.ts`, `packages/praxrr-app/src/lib/server/upgrades/processor.ts`) do the same, as do `packages/praxrr-app/src/routes/api/v1/arr/releases/+server.ts` and `packages/praxrr-app/src/routes/api/v1/arr/library/episodes/+server.ts`.
- `packages/praxrr-app/src/routes/api/v1/arr/library/+server.ts` is the gateway to the library data consumed by the `/arr/{id}/library` page, so any cache invalidation (`cache.deleteByPrefix(getArrLibraryCachePrefix(id))`) and sorting/filtering logic still talks only to `instance.url` and will continue to do so even after `external_url` is added.
- `packages/praxrr-app/src/routes/api/v1/arr/cleanup/+server.ts` and `packages/praxrr-app/src/routes/arr/[id]/logs/+page.server.ts` materialize cleanup/log actions by calling Arr APIs through clients built with `instance.url`. Those code paths should remain untouched when adding the `external_url` fallback for UI navigation.

## Configuration

- The `InstanceForm` component (`packages/praxrr-app/src/routes/arr/components/InstanceForm.svelte`) currently exposes inputs for `name`, `type`, `URL`, `API key`, `enabled` status, and `tags`. It pre-populates the dirty store with `instance.url` and writes through the hidden `<form id="save-form">` invoked by the `handleSave` helper, so introducing an `External URL` input will require adding the new field to both the dirty store (`initEdit`) and the hidden form data.
- Validation relies on `arr/test` (`packages/praxrr-app/src/routes/arr/test/+server.ts`) to talk to `instance.url`, so any external URL override must stay optional and purely presentational so that backend connectivity checks remain unchanged (`docs/plans/external-url/research-technical.md`).
- `generalSettingsQueries.shouldApplyDefaultDelayProfiles()` in the creation action (`packages/praxrr-app/src/routes/arr/new/+page.server.ts`) ensures Radarr/Sonarr instances get the correct default delay profile immediately after creation. This behavior is orthogonal to the external URL change but underscores that the database row drives both UI and job configuration, so augmenting the row with `external_url` naturally extends to delay/sync/publishing workflows.
- Existing `arr_instances` rows should default to `external_url = NULL` (per `docs/plans/external-url/research-technical.md`), so we can add the column via an `ALTER TABLE` migration (`packages/praxrr-app/src/lib/server/db/migrations/0XX_add_arr_instances_external_url.ts`) without affecting past data. All UI surfaces should compute a helper such as `const openUrl = instance.external_url?.trim() || instance.url` before using it in `window.open` or in a direct `<a>` so installations that never set the override continue to work.
