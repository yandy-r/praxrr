# Architecture Research: external-url

## System Overview

Profilarr stores each Radarr/Sonarr/Lidarr instance row in `arr_instances` (`src/lib/server/db/schema.sql` plus `src/lib/server/db/queries/arrInstances.ts`). `url`, `api_key`, `tags`, `enabled` (and eventually the new optional `external_url`) are saved there and exposed to every Arr-facing route through the `arrInstancesQueries` helpers. The `/arr` landing page (`src/routes/arr/+page.server.ts`) loads all rows to render the instance list and shares the selected row via `redirect`/`actions` (delete/cleanup). Once an instance is created or updated, SvelteKit re-runs `src/routes/arr/[id]/+layout.server.ts`, so every nested route (`settings`, `library`, `logs`, etc.) has the latest row via `instance = arrInstancesQueries.getById(id)`.

`InstanceForm` (`src/routes/arr/components/InstanceForm.svelte`) is the single UI surface that feeds the `?/update` and `?/default` actions defined in `src/routes/arr/[id]/settings/+page.server.ts` and `src/routes/arr/new/+page.server.ts`. Those actions validate/normalize the form data (currently `name`, `type`, `url`, `api_key`, `tags`, `enabled`), call `arrInstancesQueries.update/create`, and log outcomes; they also trigger job cleanup (`cleanupJobsForArrInstance` in `src/lib/server/jobs/cleanup.ts`) when a row is deleted. The settings form and the library page both render with `instance` data provided by the layout load, so any new per-instance metadata appears immediately after the action completes.

The library tab (`src/routes/arr/[id]/library/+page.svelte`) is a client-heavy view. It derives `instanceUrl` from `instance.url`, keeps pagination/search state in `$page` plus `getPersistentSearchStore`, caches the last API result in `libraryCache` (`src/lib/client/stores/libraryCache.ts`), and fetches data from `GET /api/v1/arr/library` (`src/routes/api/v1/arr/library/+server.ts`) for the currently selected instance. The API route talks to the Arr app through `RadarrClient`, `SonarrClient`, or `LidarrClient` (`src/utils/arr/clients/*.ts`), validates query params, and stores results in the shared `cache` helper (`src/lib/server/utils/cache/cache.ts`) so repeated access benefits from a 5‑minute TTL. The UI uses `instanceUrl`/`baseUrl` to build the “Open in” button, the toolbar link, and the movie/series/artist links rendered in `ExpandableTable` rows, while `LibraryActionBar.svelte` and the `TableView`/`CardView` components expose explicit `ExternalLink` buttons that currently call `window.open(instance.url, '_blank')`.

## Relevant Components

- `src/routes/arr/components/InstanceForm.svelte`: Central form that collects `name/type/url/api_key/tags/enabled`, tracks dirty state, and submits through the hidden `/save-form`/`/delete-form`. A new `External URL` input and hidden `<input name="external_url" ...>` need to tie into this form so existing actions receive the value.
- `src/routes/arr/new/+page.server.ts` & `src/routes/arr/[id]/settings/+page.server.ts`: Create/update actions parse `FormData`, validate (via `arrInstancesQueries.nameExists/apiKeyExists`), and call `arrInstancesQueries.create/update`. They are the only places new instance metadata (including future `external_url`) is persisted and logged, and the settings action is responsible for keeping the delete flow tied to `cleanupJobsForArrInstance`.
- `src/lib/server/db/queries/arrInstances.ts`: SQL helper that inserts, updates, deletes, and queries `arr_instances`. It currently marshals `tags` to JSON and enforces `enabled` default semantics; adding `external_url` means updating `ArrInstance`, `CreateArrInstanceInput`, `UpdateArrInstanceInput`, and the SQL statements so `NULL`/empty strings are normalized.
- `src/lib/server/db/schema.sql`: Canonical schema describing the `arr_instances` table and all downstream foreign key dependencies (jobs, release syncs, caches). The new `external_url TEXT NULL` column will live here and feed every migration+seed script referencing the table.
- `src/routes/arr/+page.svelte` plus `src/routes/arr/views/{TableView,CardView}.svelte`: List view that renders `instances` returned by `+page.server.ts`. Both `ExternalLink` buttons call `window.open(instance.url, '_blank')`, and `Badge` elements show `instance.url`, so any display logic must consider the new `external_url` fallback.
- `src/routes/arr/[id]/library/+page.svelte`: Uses the `instance` from layout, keeps pagination/search state in `$page`, fetches `/api/v1/arr/library`, caches responses, and derives `instanceUrl`/`baseUrl` for toolbar and row links. The `handleOpen` helper and the `baseUrl` used by the action bar and table rows consume this URL directly.
- `src/routes/arr/[id]/library/components/LibraryActionBar.svelte`: Contains the `Open in` dropdown button (calls `onOpen`, which the page passes `handleOpen`), the refresh dropdown (fires `handleRefresh` which invalidates both client `libraryCache` and server `cache` via `DELETE /api/v1/arr/library`), and exposes pagination/filters for the table.
- `src/routes/api/v1/arr/library/+server.ts`: Server handler that validates `instanceId`, `page`, `pageSize`, `sortKey`, and `sortDirection`, checks the cached payload via `buildArrLibraryCacheKey`, obtains profile names from `pcdManager` (`src/lib/server/pcd/index.ts`) through `qualityProfileQueries` (`src/lib/server/pcd/entities/qualityProfiles/index.ts`), and delegates to the appropriate Arr client (`RadarrClient`, `SonarrClient`, `LidarrClient`) to pull the library.
- `src/lib/client/stores/libraryCache.ts`: Client-side cache keyed on `instanceId` plus pagination/search/sort parameters. The library page reads it before refetching and invalidates it (and the server `cache`) when the user hits refresh, keeping UI/performance consistent even while the backend is throttled.
- `src/routes/arr/test/+server.ts`: Connection-test endpoint used by `InstanceForm` (via `fetch('/arr/test'...)`). It keeps using `url` (not `external_url`), so server-side flows remain bound to the internal API endpoint even after the frontend introduces a separate browser-friendly URL.

## Data Flow

- User navigates to `/arr/new` or `/arr/{id}/settings` and sees `InstanceForm` (`src/routes/arr/components/InstanceForm.svelte`), which seeds `initEdit` with the current `instance` values and wires hidden `<form>` elements to `?/default` or `?/update` via `use:enhance`.
- Clicking Save triggers `handleSave` → test connection POST to `/arr/test` (`src/routes/arr/test/+server.ts`) using the `url`/`api_key`, and when success it calls `saveForm.requestSubmit()` so SvelteKit runs either `src/routes/arr/new/+page.server.ts` or `src/routes/arr/[id]/settings/+page.server.ts`.
- Those actions re-parse the trimmed form data, re-validate duplicate names/API keys (`src/lib/server/db/queries/arrInstances.ts`), and call `create`/`update` to persist the row; updates also log via `logger` and run `cleanupJobsForArrInstance` when deleting (`src/lib/server/jobs/cleanup.ts`).
- After the action completes, SvelteKit reloads `src/routes/arr/[id]/+layout.server.ts` for that instance, re-running `arrInstancesQueries.getById` so every child route receives the latest `instance` object (with the new `external_url` field, once added).
- `/arr` (`src/routes/arr/+page.server.ts`) keeps a `load` that returns `arrInstancesQueries.getAll()`; the list view uses `createDataPageStore` to manage search/view state and renders either `TableView` or `CardView`, both of which use `resolve('/arr/[id]')` for instance links and `window.open(instance.url, '_blank')` for the “Open in” buttons.
- `/arr/{id}/library/+page.svelte` reads the layout `instance`, strips the trailing slash (`instance?.url?.replace(/\/$/, '')`), and assigns it to `instanceUrl`/`baseUrl`. It fetches `/api/v1/arr/library?instanceId={instance.id}&page=...` to retrieve cached/paginated data, stores it locally in `libraryCache`, and renders `LibraryActionBar`, `ExpandableTable`, and other row components that reference `baseUrl` when building `/movie/{tmdbId}` / `/series/...` / `/artist/...` links.
- The toolbar’s “Refresh” dropdown calls `handleRefresh`, which invalidates the client `libraryCache`, hits `DELETE /api/v1/arr/library?instanceId={instance.id}` (the API handler just calls `cache.deleteByPrefix(getArrLibraryCachePrefix(id))`), and re-fetches the library, ensuring both client and server caches stay in sync.
- The backend `GET /api/v1/arr/library` (`src/routes/api/v1/arr/library/+server.ts`) validates parameters, fetches the instance row via `arrInstancesQueries.getById`, grabs the relevant profile names from every enabled PCD (`pcdManager` + `qualityProfileQueries`), and calls the Arr client (Radarr/Sonarr/Lidarr) to materialize the library before applying pagination/sorting and returning the payload.

## Integration Points

- **`InstanceForm` (`src/routes/arr/components/InstanceForm.svelte`)**: Add a visible `External URL` input (optional, with descriptive helper text) whose `on:input` updates the dirty store, plus a hidden `<input name="external_url" ...>` in the `save-form` so both `?/default` and `?/update` receive the value. Trim/normalize the value the same way `url` is trimmed and ensure dirty tracking resets whenever the action reports success.
- **Actions (`src/routes/arr/new/+page.server.ts`, `src/routes/arr/[id]/settings/+page.server.ts`)**: Parse `external_url` from `FormData`, validate that a non-empty value is an absolute `http(s)` URL (per `docs/plans/external-url/feature-spec.md`), normalize empty strings to `null`, and pass it into `arrInstancesQueries.create/update`. Keep server flows (job cleanup, logging, duplicate-name/API key checks) unchanged; the new field only affects a browser link.
- **`arrInstancesQueries` (`src/lib/server/db/queries/arrInstances.ts`)**: Extend `ArrInstance`, input types, and the SQL insert/update statements so `external_url` is returned, inserted, and updated. Treat empty/whitespace `external_url` as `NULL` (similar to how `tags` is serialized) and ensure `update` pushes `updated_at = CURRENT_TIMESTAMP` when the field changes.
- **Layout load (`src/routes/arr/[id]/+layout.server.ts`)**: Already exposes the `instance` row to every nested route. Once `arrInstancesQueries.getById` returns `external_url`, the UI components rendered under this layout automatically get the new field without additional subscriptions, so they can compute `openUrl = instance.external_url ?? instance.url` once per render.
- **Library page (`src/routes/arr/[id]/library/+page.svelte` & `LibraryActionBar.svelte`)**: Replace `instanceUrl = instance?.url?.replace(/\/$/, '')` with a helper like `const instanceOpenUrl = (instance?.external_url?.trim() || instance?.url)?.replace(/\/$/, '')`. Use that `instanceOpenUrl` in `handleOpen`, `baseUrl`, and anywhere rows build `href`. The “Open in” toolbar button and the per-row movie/series/artist links should all reuse this computed value so toggling `external_url` switches every link at once.
- **Instance list views (`src/routes/arr/views/TableView.svelte` & `CardView.svelte`)**: Update `handleExternalClick` to accept a resolved open URL (new helper `resolveOpenUrl(instance)`). Consider showing `external_url` (when present) in the `Badge` or helper text so users know what the link will use. Ensure `window.open` uses the same fallback logic as the library page so the dropdown action and card/table buttons behave identically.

## Key Dependencies

- `docs/plans/external-url/feature-spec.md`: Defines the optional `external_url` column, the fallback `external_url || url` rule for UI links, the validation expectations, and the zero-migration-backfill requirement that existing rows default to `NULL`.
- `docs/plans/external-url/research-technical.md`: Lays out the exact files to touch (form input, create/update actions, `arrInstancesQueries`, library toolbar, etc.), emphasizes keeping all backend Arr calls on the canonical `url`, and highlights the helper/fallback pattern so multiple UI surfaces stay in sync after `external_url` changes.
