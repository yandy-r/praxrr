# Resolved Config Viewer — Security Research

Scope: GitHub issue #25 — a read-oriented `/api/v1/*` surface that (a) reads the already-compiled
PCD cache to show base/user/resolved layers per entity, (b) compares resolved state across
multiple linked Arr instances, and (c) fetches live state from one or more Arr instances to diff
against the resolved (desired) state. Findings are grounded in the current codebase
(`hooks.server.ts`, `$auth/middleware.ts`, `$arr/urlSafety.ts`, `$arr/arrInstanceClients.ts`,
`$db/queries/arrInstances.ts`, `$db/queries/arrInstanceCredentials.ts`,
`routes/api/v1/arr/library/+server.ts`, `routes/api/v1/sync/preview/+server.ts`,
`routes/databases/[id]/changes/components/FieldDiffTable.svelte`, `$utils/rateLimit.ts`,
`sync/preview/limits.ts`) as of this session, plus external research on candidate diff
libraries — not a generic checklist. This is advocacy, not gatekeeping: every finding ships with
a mitigation that reuses an existing repo primitive rather than a redesign.

## Executive Summary

The feature is fundamentally low-risk on the read side: resolved/layered config comes from the
already-compiled PCD cache (local SQLite, no new remote surface), and `hooks.server.ts` already
gates every non-`PUBLIC_PATHS` route under `/api` uniformly — a new `/api/v1/resolved/*` handler
needs zero manual auth wiring as long as it isn't added to `PUBLIC_PATHS`. Credential handling is
also solid and reusable as-is: `arrInstancesQueries` never selects the plaintext `api_key` column
(`'' AS api_key` is hard-coded into every `SELECT`), and the real key only exists in memory
briefly inside `getArrInstanceClient()` after AES-GCM decryption.

The risk is concentrated in the two *new* surfaces this feature adds beyond what already exists:
(1) **live/cross-instance diff rendering**, which is exactly the kind of view the codebase's one
existing diff component (`FieldDiffTable.svelte`, used by `databases/[id]/changes`) already
renders via unsanitized `marked.parse()` + `{@html}` for `description`/`notes`/`readme`-shaped
fields — reusing that pattern for config sourced from third-party PCD repos or live Arr responses
is a real stored-XSS vector; and (2) **fan-out live fetches across N instances**, a genuinely new
request-amplification shape (today's `sync/preview` and `arr/library` endpoints only ever hit one
instance per request) that needs its own rate limit and instance-count bound, and that inherits a
pre-existing gap where the codebase's own SSRF guard (`assertSafeArrUrl`) is wired into only two
"Test Connection" endpoints, not into the actual fetch choke point (`getArrInstanceClient`) that
this feature — like `arr/library` and `sync/preview` today — will call.

None of this requires new architecture. Every mitigation below is "call the helper that already
exists" or "don't call `marked.parse()` directly."

## Findings by Severity

### CRITICAL

| ID | Finding | Risk | Mitigation |
| --- | --- | --- | --- |
| C1 | The codebase's only existing diff-rendering component, `FieldDiffTable.svelte` (`routes/databases/[id]/changes/components/FieldDiffTable.svelte:196-198,250-252`), calls `marked.parse()` directly and renders the result via `{@html}` for any field named `description`/`readme`/`notes`, with **no** sanitization step. The client-wide `Markdown.svelte` component has the identical gap (`marked.parseInline`/`marked.parse` → presumably `{@html}`, no sanitizer). A properly sanitized `parseMarkdown()` (allowlisted tags/attrs, strips `<script>`, `on*=`, `javascript:` URLs) already exists at `$server/utils/markdown/markdown.ts:93-101` but neither component calls it. Resolved-config-viewer's layer-breakdown and diff-against-live views are the natural extension point for this exact component shape (`FieldRow`/`DraftEntitySection`, already generic across custom formats, quality profiles, media management, naming) and will very likely reuse it or copy its pattern for entity `description`/notes-style fields. | If reused as-is, any text field containing markdown-passthrough HTML — sourced from a linked PCD git repo (semi-trusted, often third-party/community) or, worse, from a **live Arr instance API response** in the diff-against-live view (an Arr instance the operator doesn't fully control, or one reached via a spoofed/compromised URL) — executes as script in the authenticated operator's session when the resolved/diff view renders it. This is a standard stored-XSS-to-session-takeover path (session cookie theft, silent instance/PCD mutation via the same session). | Do not call `marked.parse()`/`marked.parseInline()` directly in any new diff/tree component. Route every markdown-bearing field through the existing sanitized `parseMarkdown()` in `$server/utils/markdown/markdown.ts`, computed server-side and shipped to the client as pre-sanitized HTML (or, simpler and safer for a v1: render all resolved/diff values as plain escaped text — Svelte's default `{value}` interpolation — and skip markdown rendering in the diff view entirely; markdown rendering is a nice-to-have for descriptions, not a requirement for a resolved-state/diff viewer). |

### WARNING

| ID | Finding | Risk | Mitigation |
| --- | --- | --- | --- |
| W1 | `assertSafeArrUrl()` (`$arr/urlSafety.ts`) — the only SSRF guard in the codebase (blocks cloud metadata IPs and link-local ranges; intentionally allows RFC1918/loopback for LAN Arr instances) — is called from exactly two places: `routes/api/v1/setup/test-connection/+server.ts` and `routes/arr/test/+server.ts`. | W2 | `routes/api/v1/arr/library/+server.ts:489-497` (the closest existing precedent for "fetch live Arr state and return it") returns the raw `err.message` from a failed `getArrInstanceClient()`/HTTP call straight into the JSON error response body (`{ error: message }`). `HttpError`/network-failure messages routinely embed the target host/port (e.g. `connect ECONNREFUSED 10.0.0.5:7878`). | W3 | Cross-instance comparison and live-diff are inherently fan-out: one UI action can trigger fetches against an arbitrary number of instances (`arrInstancesQueries.getAll()`/`getEnabled()` has no upper bound). Existing precedent for a *single*-instance fetch (`sync/preview`) already has both a rate limiter (`registerPreviewCreateAttempt()`, `sync/preview/limits.ts`, 6/60s per instance) and a store-size cap (`PREVIEW_MAX_SNAPSHOTS`); `arr/library` caches per-instance for 5 minutes. No equivalent multi-instance bound exists anywhere in the codebase today because no current feature fans out to more than one instance per request. | Without a bound, a single API call can become N simultaneous outbound HTTP requests to N Arr instances (self-inflicted DoS against those instances) and N credential-decrypt operations server-side; repeated calls amplify further since there's no per-request cap today to reuse. | Add a dedicated limiter module mirroring `sync/preview/limits.ts` (e.g. `resolved/limits.ts`): (a) a `registerRateLimitAttempt()`-style window per requesting user/IP (reuse `$utils/rateLimit.ts` directly — it's already generic, keyed by arbitrary string), and (b) a hard cap on `instanceIds.length` per comparison/live-diff request (mirroring `LIBRARY_MAX_PAGE_SIZE`'s "clamp, don't error" pattern, e.g. cap at 10-20 instances per call with a clear 400 above that). |
| W4 | Resolved-config-viewer needs `entityId`/`entity` (custom format, quality profile, etc.) plus `arr_type` to resolve which cache/entity table to read and which Arr-specific mapping applies. The codebase enforces per-Arr-type validation inconsistently across call sites — some use a canonical `isArrAppType()` (`$shared/arr/capabilities.ts`), others a local `isSyncPreviewArrType()` (`sync/preview/+server.ts`) or a hardcoded `VALID_TYPES` array (`createInstanceAction.ts`) — and CLAUDE.md's own "Cross-Arr Semantic Validation Policy" explicitly requires "Read/write/sync dispatch resolves by explicit `arr_type` (no implicit sibling fallback)" and fail-fast on ambiguous mappings. | An entity/arr_type combination that's accepted without validating the pairing (e.g. a Lidarr-shared-Sonarr-table entity ID looked up under the wrong `arr_type`) can silently return resolved config computed under the wrong app's semantics — a data-integrity/authorization-adjacent bug (cross-Arr data confusion), not classic injection, but concretely a *wrong data disclosed* risk specific to this codebase's shared-table transitional patterns (Sonarr-backed Lidarr entities, noted in CLAUDE.md). | Validate `arr_type` against one canonical allowlist (`isArrAppType()`) at the top of every new resolved-config handler, and validate that the resolved `entityId` actually belongs to the requested `arr_type` before reading/returning it (reuse the same quality-name/mapping validation pattern CLAUDE.md mandates for quality-profile compatibility) — fail fast (400) on any ambiguous pairing rather than falling back to a sibling app's semantics. |
| W5 | Any new query built for "list arr_instances for the comparison picker" must go through `arrInstancesQueries` (`getAll()`/`getEnabled()`/`getByType()`), all of which hard-code `'' AS api_key` in their `SELECT`. There's no evidence anyone will bypass this, but it's a foot-gun: a fresh, ad-hoc `SELECT * FROM arr_instances` written for this feature's new instance-picker UI would trivially reintroduce plaintext-key exposure that `arrCredentialRedactionRoutes.test.ts` exists specifically to catch for other routes. | If bypassed, instance list responses used to populate the comparison/live-diff UI would leak `api_key` (or `api_key_fingerprint`, lower severity) to the client. | Reuse `arrInstancesQueries` accessors exclusively for any instance metadata the new feature needs; add the new endpoint(s) to the existing `arrCredentialRedactionRoutes.test.ts` suite (it's designed to be extended per-route) rather than writing a bespoke test. |

## Authentication and Authorization

- **Are new `/api/v1` resolved-state endpoints covered by existing auth middleware?** Yes, by construction. `hooks.server.ts:208-266` is a single global `handle` that runs before every request; it 401s any `/api`-prefixed path when `!auth.user` and `!auth.skipAuth`, and the public-path allowlist (`PUBLIC_PATHS` in `$auth/middleware.ts:27`, currently `['/auth/login', '/auth/setup', '/auth/oidc', '/api/v1/health']`) is a short, explicit list. A new `/api/v1/resolved/*` (or however it's namespaced) route needs **zero** manual auth code as long as nobody adds it to `PUBLIC_PATHS` — unlike frameworks where routes opt in to middleware, here every route is covered unless explicitly excluded. Recommend a regression test asserting 401-without-session on the new endpoints (mirrors the discipline `arrCredentialRedactionRoutes.test.ts` already applies to other routes) as a tripwire against a future accidental `PUBLIC_PATHS` addition, not because a gap exists today.
- **Risk of exposing config without auth when `AUTH=off`:** see A1 — `AUTH=off` is a supported reverse-proxy-trust mode, not dev-only, so this is pre-existing accepted risk rather than a new gap; worth reiterating in this feature's own docs given resolved/live-diff surfaces more operationally-sensitive detail (naming conventions, scoring logic, LAN topology) than most existing screens.
- **`AUTH=local` (bypass on local IP):** identical exposure model to `AUTH=off` for LAN-based requests — same reasoning applies; not a new risk introduced by this feature.

## Data Protection

- Arr instance API keys are never returned by any `arrInstancesQueries` read — every `SELECT` in `arrInstances.ts` hard-codes `'' AS api_key`; the real key exists only encrypted at rest (`arr_instance_credentials`, AES-GCM, versioned keys — see `$server/utils/encryption/arr-credentials.ts`) and briefly in memory inside `getArrInstanceClient()` after decryption, scoped to the request. This feature should read instance metadata exclusively through `arrInstancesQueries` (see W5) and never construct a fresh raw query.
- Live-diff responses (Arr instance state fetched for comparison) must not echo back any credential-shaped value the Arr API itself might return (Arr's own `/system/status` and config endpoints don't return the caller's own API key, so this is low-risk, but any new response shape should be explicitly typed — like `LibraryResponse` in `arr/library` — rather than passing through raw upstream JSON, both for safety and to avoid accidentally forwarding an unexpected field).
- Resolved config itself (custom format regex patterns, scoring, naming templates) is not a credential, but it is the operator's curated "IP" in a piracy-adjacent-detection sense — worth keeping behind the same auth boundary as everything else (already true, see Authentication and Authorization) rather than treating it as low-sensitivity.
- No plaintext-credential leak path was found in any endpoint reviewed (`arr/library`, `sync/preview`); the existing `arrCredentialRedactionRoutes.test.ts` suite should be the model for testing the new endpoints (W5).

## Input Validation

- **Instance IDs:** existing precedent (`arr/library:307-321`) is `parseInt` + `isNaN` check + `arrInstancesQueries.getById()` existence check before any fetch — replicate this exactly for every `instanceId` this feature accepts, including each ID inside a cross-instance-comparison array (validate each element, not just array shape).
- **Entity IDs / `arr_type`:** see W4 — validate `arr_type` against the canonical `isArrAppType()` allowlist and confirm the entity actually belongs to that `arr_type` before returning resolved data for it; fail fast per CLAUDE.md's Cross-Arr Semantic Validation Policy rather than falling back to a sibling app's mapping.
- **SSRF (server fetches from user-configured instance URLs):** the instance URL itself is not attacker-supplied per-request (it's `instanceId` → a stored, previously-configured URL), which bounds this feature's *direct* exposure — but it fully inherits whatever protection (or lack of it) exists on the stored URL, which today is only enforced at "Test Connection" time, not at actual-fetch time (W1). Reusing `getArrInstanceClient()` (as this feature should) is correct and sufficient *once* W1's fix (guard inside `getArrInstanceClient`) lands; until then, this feature inherits the same gap every other Arr-fetching endpoint already has.
- **XSS in diff/tree rendering:** see C1 — the concrete risk is markdown-shaped fields (`description`/`notes`/`readme`) rendered via unsanitized `{@html}`. Release-title regex/pattern-shaped fields are lower risk: the existing `FieldDiffTable.svelte` renders `pattern`/naming-template fields as plain text inside `Badge` components (`{beforeValue}`/`{afterValue}` — Svelte's default escaped interpolation, not `{@html}`), which is already the safe pattern and should be the default for *all* resolved-config field types unless a field is explicitly markdown-shaped and explicitly sanitized.
- **SQL injection:** no gap found (A3) — continue parameterized queries / Kysely for any new tables or read paths this feature introduces.

## Infrastructure Security

- **Rate limiting for fan-out live fetches:** no bound exists today for multi-instance requests (W3) because no existing feature fans out beyond one instance per call — this is new territory this feature specifically introduces, and needs both a request-rate limiter (reuse `$utils/rateLimit.ts`) and a per-request instance-count cap.
- **Error messages that could leak internal URLs:** `arr/library`'s existing pattern of forwarding `error.message` verbatim (W2) is the anti-pattern to avoid; `testConnectionReason.ts` is the existing sanitized-reason-enum precedent to reuse instead.
- **CORS:** none configured anywhere (A2) — preserve that for the new endpoints.
- **Caching:** if this feature caches live-diff results, key by layer/entity in addition to instanceId (A4) to avoid serving stale/wrong-layer data.

## Secure Coding Guidelines

1. No manual auth check is required in new handlers (global middleware covers `/api` by default) — but never add the new routes to `PUBLIC_PATHS`, and add a 401-without-session regression test per route (mirrors `arrCredentialRedactionRoutes.test.ts`).
2. Read instance metadata exclusively through `arrInstancesQueries` accessors — never a fresh raw `SELECT * FROM arr_instances` (W5).
3. Fetch live Arr state exclusively through `getArrInstanceClient()` — never construct a client or call `fetch()` directly against a stored instance URL (W1).
4. Do not render any field via `{@html}`/`marked.parse()` in new diff/tree components without going through the existing sanitized `parseMarkdown()` (`$server/utils/markdown/markdown.ts`); default to plain escaped-text rendering for resolved/diff values unless a field is explicitly markdown-shaped (C1).
5. Never forward `error.message`/raw `HttpError` text into a JSON error response for live-diff/comparison failures — map to a sanitized reason enum (reuse/extend `testConnectionReason.ts`) and log full detail server-side (W2).
6. Validate `arr_type` against `isArrAppType()` (or an equivalently canonical allowlist) at the top of every handler; validate that `entityId` actually belongs to the requested `arr_type` before returning resolved data for it (W4).
7. Any cross-instance/live-diff fan-out must call a rate limiter (`$utils/rateLimit.ts` or a dedicated `resolved/limits.ts` mirroring `sync/preview/limits.ts`) and enforce a hard cap on instances-per-request (W3).
8. Keep all new queries parameterized (Kysely or `?`-placeholder SQL) — no string-interpolated SQL (A3).

## Trade-off Recommendations

- **Markdown rendering in the diff view:** default to plain escaped-text rendering for v1 (zero new sanitization surface, consistent with how `pattern`/naming-template fields already render safely today) rather than reusing `FieldDiffTable.svelte`'s unsanitized markdown path as-is. Rich markdown formatting for `description`/`notes` fields can follow once the sanitizer gap (C1/A5) is fixed centrally — don't let this feature be the second place that copies the unsafe pattern while waiting on that fix.
- **Diff library:** hand-roll the base/user/resolved three-way diff against the PCD entity's known TypeScript shape (primary recommendation — zero new dependency, matches the existing `databases/[id]/changes` precedent, and the repo's own dependency-hygiene convention requires justifying any new dependency against what already exists). Fall back to `microdiff` only if the comparison logic genuinely needs generic recursive diffing; avoid `jsondiffpatch` specifically because of its CVE history and its HTML-formatter attack surface, which runs directly counter to closing C1.
- **SSRF guard placement:** prefer fixing this once, centrally, inside `getArrInstanceClient()` (W1) over adding a duplicate `assertSafeArrUrl()` call only inside this feature's new endpoint — the centralized fix benefits `arr/library`, `arr/releases`, `sync/preview`, and upgrades simultaneously with one small, low-risk change, whereas a feature-local fix leaves every other existing fetch path (and the next new feature) still exposed. If time-boxed, at minimum add the call in this feature's own fetch path and file a follow-up issue for the centralization.
- **Rate-limit granularity:** apply both a per-request instance-count cap (protects downstream Arr instances from a single oversized comparison request) and a per-user/IP request-rate window (protects Praxrr itself from repeated large requests) — the two existing precedents (`sync/preview/limits.ts` per-instance, `$utils/rateLimit.ts` generic-key) are complementary, not substitutes, and this feature is the first to need both simultaneously.

## Open Questions

1. Should cross-instance comparison be restricted to same-`arr_type` instances only (e.g., never compare a Radarr instance's resolved state against a Sonarr instance's), or is cross-type comparison a deliberate use case? This affects both the W4 validation design and the UX for the instance picker.
2. What is the intended trust level of markdown-bearing fields (`description`/`notes`/`readme`) sourced from a *linked PCD source* (which may be a third-party/community git repo) versus fields sourced from a *live Arr instance response*? Both currently share the same unsanitized rendering path in `FieldDiffTable.svelte`; the resolved-config-viewer may want to treat them differently (e.g., PCD-sourced text as semi-trusted markdown once sanitized, live-Arr-sourced text as always-plain-text since Arr's own API isn't expected to author markdown).
3. Is the centralized `assertSafeArrUrl()`-in-`getArrInstanceClient()` fix (W1) in scope for this feature's implementation, or should it be tracked as a separate, prerequisite security fix given it touches every existing Arr-fetching endpoint, not just the new ones?
4. What's the expected maximum instance count for "cross-instance comparison" in realistic deployments — this determines whether the W3 fan-out cap should be a small fixed number (e.g., 10) or configurable.
5. Should live-diff results ever be cached, and if so, for how long — Arr instance state can change between an operator's individual syncs, so a stale cached "live" value defeats the feature's purpose of showing *current* drift; if caching is added purely for repeated-view performance, the TTL should likely be much shorter than `arr/library`'s existing 5-minute TTL.
