# Praxrr Parser Go Migration — Engineering Practices Research

## Executive Summary

Issues #1–#5 should be treated as a compatibility migration, not a greenfield
parser rewrite. The smallest maintainable design is one Go module at
`packages/praxrr-parser`, one executable under `cmd/praxrr-parser`, and a small
number of internal packages: a single cohesive parser package, a wire-contract
package, an HTTP adapter, and the executable composition root. Port the existing
decision order and regexes literally first, prove observable parity, and only
then make idiomatic refactors that preserve the corpus.

The largest risk is not basic Go syntax. The current parser depends on
.NET-specific behavior:

- lookarounds, backreferences, atomic groups, inline option changes,
  ignore-pattern-whitespace, and repeated captures of the same named group;
- `Regex.Matches`, last-match/last-capture selection, and .NET string/case
  behavior;
- precise JSON defaults such as `[]` rather than `null`, enum names rather than
  numeric values, and nullable movie/episode fields;
- endpoint validation text and status codes;
- dictionary semantics that collapse duplicate patterns and duplicate texts;
- a 100 ms timeout for caller-supplied `/match` regexes, but no timeout on the
  static domain parsers;
- a parser version that participates in application cache invalidation.

`regexp2` is the justified dependency because its default dialect is based on
the .NET engine and it supports the constructs used here. It must be wrapped
behind a very small repository-owned adapter. Do not use its `RE2` mode: that
changes `\d`, `\s`, `\w`, and `$` semantics. Do not accept its timeout, capture,
or backtracking defaults without explicit tests. The current `regexp2/v2` line
is also materially different from v1: it requires Go 1.25+, adds a default
100,000-slot backtracking stack bound, and exposes explicit capture-order and
memory options.

The merge gate should be a checked-in, deterministic fixture corpus generated by
the legacy .NET service plus a temporary differential test that runs the same
requests against both binaries. The Go implementation must never regenerate its
own expected outputs. Delete the .NET source only after unit, golden,
differential HTTP, race, Docker, standalone-binary, and cross-platform build
gates pass.

## Modularity Design

Use one module because the parser is built, versioned, released, and removed as
one unit. A nested Go module is appropriate here: it prevents root Deno tooling
from accidentally treating Go dependencies as part of the application module,
while `go test ./...` remains scoped and fast.

```text
packages/praxrr-parser/
  go.mod
  go.sum
  cmd/praxrr-parser/
    main.go                  # flags/env, logger, HTTP server, signals, build version
  internal/contract/
    request.go               # exact JSON request shapes
    response.go              # exact JSON response shapes and enum-to-wire names
  internal/parser/
    regex.go                 # the only regexp2 adapter
    common.go                # file-extension and title cleanup
    quality.go
    language.go
    releasegroup.go
    title.go
    episode.go
    service.go               # compose one parse result, no HTTP knowledge
  internal/httpserver/
    handler.go               # routes, decode/validate/encode, request logging
    handler_test.go
  testdata/
    parity/                  # immutable input + .NET-produced expected output
    regex/                   # invalid, timeout, repeated-capture, Unicode cases
```

Keep all five domain parsers in one `parser` package initially. They share
cleanup rules, regex semantics, models, and orchestration, and they are not
independent public libraries. Creating `quality`, `language`, `title`,
`episode`, `releasegroup`, `regexutil`, `models`, and `common` packages would
reproduce the C# directory layout as package ceremony, force exports, and create
dependency edges without improving isolation. Split only when a package has an
independently useful API and a clear ownership boundary.

Use `contract`, not `models` or `types`, because these structs specifically
represent the wire contract. Use `httpserver`, not a generic `api` package. The
command should contain no parsing logic; it should construct a parser service,
register handlers, configure server timeouts, and handle signals.

Recommended dependency direction:

```text
cmd/praxrr-parser -> internal/httpserver -> internal/parser -> internal/contract
                                      \----> internal/contract
```

`parser` must not import `net/http`, environment variables, logging globals,
Docker concerns, or the TypeScript application. `httpserver` should depend on
one narrow service interface only if that interface directly enables handler
tests; define the interface at the consumer (`httpserver`), not preemptively in
the parser package.

### Transliterate first, refactor second

Transliterate branch order, early returns, parsing defaults, and capture
selection closely enough that a reviewer can compare C# and Go side by side.
This is one of the rare cases where a deliberately literal first port is safer
than an immediately “clean” design. Examples:

- Preserve `QualityParser`'s early-return precedence. A table-driven “rules
  engine” is more elegant only if it demonstrably preserves all overlapping
  source/modifier cases.
- Preserve `LanguageParser` insertion order followed by stable distinct
  filtering. A Go map used as a set must not become the output iterator, because
  map iteration is not the current ordering contract.
- Preserve last match and last capture behavior. `regexp2` exposes every
  repeated capture in `Group.Captures`; `Group.String()` represents only the
  last capture.
- Preserve the episode regex list order and range expansion semantics. Do not
  merge similar regexes during migration.
- Preserve catch-all domain behavior at the public service boundary (`null`
  episode/title outcome) while replacing internal swallowed errors with
  contextual Go errors and tests.

After the golden and differential suites are green, refactor duplicated helper
mechanics in small commits with the same suite running. Do not opportunistically
“fix” legacy parsing quirks inside issues #1–#5; record a fixture and open a
follow-up behavior-change issue.

## Regex Compatibility Practices

Create exactly two compilation policies in `internal/parser/regex.go`:

1. **Static parser patterns:** source-controlled patterns compiled at process
   startup. A compile failure is a programmer/build failure and should stop
   startup with the pattern's stable name.
2. **Caller patterns:** untrusted `/match` patterns compiled per distinct
   submitted pattern. Compile, timeout, and backtracking errors map to `false`,
   matching the existing endpoint.

Do not scatter direct `regexp2.Compile`, `MatchString`, `FindStringMatch`,
`FindNextMatch`, or replacement calls across domain files. The wrapper should:

- translate only the actually used options: default .NET mode, `IgnoreCase`, and
  `IgnorePatternWhitespace`; `RegexOptions.Compiled` has no semantic Go
  equivalent and should be omitted;
- expose named-group lookup, all captures in order, all matches in order, match,
  and replacement;
- always inspect returned errors;
- retain a stable regex identifier in errors without logging arbitrary pattern
  contents at high severity;
- document rune offsets. Prefer `Capture.String()` rather than slicing the
  original UTF-8 string;
- avoid the `compat` adapter because its no-error signatures panic on
  timeout/stack-limit failures;
- never enable `regexp2.RE2`, `ECMAScript`, or `Unicode` modes without a
  separately approved parity change.

Example adapter surface (illustrative, not a required abstraction hierarchy):

```go
type expression struct {
    name string
    re   *regexp2.Regexp
}

func compileStatic(name, pattern string, options regexp2.RegexOptions) (expression, error)
func (e expression) match(input string) (bool, error)
func (e expression) first(input string) (*regexp2.Match, error)
func (e expression) all(input string) ([]*regexp2.Match, error)
func (e expression) replace(input, replacement string) (string, error)
```

Compile every static pattern in a dedicated test even though startup also
compiles them. Add focused compatibility fixtures for constructs visibly present
in this repository:

- `.NET` named groups `(?<name>...)` and named backreferences `\k<name>`;
- repeated `season`, `episode`, `absoluteepisode`, `version`, and `R2160p`
  captures;
- positive/negative lookbehind, including the nontrivial lookbehinds already in
  title/quality patterns;
- atomic group `(?>...)` in mini-series matching;
- `IgnorePatternWhitespace` plus inline `(?i)` and `(?-i:...)` option changes;
- `\b`, `\w`, `\W`, `\d`, and case folding across ASCII, accented Latin, CJK,
  supplementary Unicode, and malformed UTF-8 decoded through JSON;
- replacement syntax and empty matches;
- multiple matches and last-match selection.

### Backtracking and timeout decision

Strict parity and hardening pull in different directions. Current .NET behavior
is:

- `/match` and `/match/batch`: 100 ms `Regex` timeout;
- static quality/language/title/episode/release-group regexes: no explicit
  timeout;
- no regexp2-style 100,000-slot stack limit.

`regexp2/v2` now bounds its interpreter stack by default and checks
`MatchTimeout` with a process-wide background clock whose default period is
approximately 100 ms. Therefore, simply setting
`MatchTimeout = 100*time.Millisecond` is not proof of equivalent timing.

For the parity release:

- explicitly choose the backtracking-stack policy rather than inheriting a
  library default;
- preserve 100 ms on user patterns and classify both timeout and
  `ErrBacktrackingStackLimit` as `false`;
- set `regexp2.SetTimeoutCheckPeriod(...)` once, before any timed matches, then
  use deterministic tests with deliberately catastrophic patterns and generous
  scheduling tolerances;
- call `regexp2.StopTimeoutClock()` from test teardown when tests inspect
  goroutine lifetimes;
- benchmark representative and adversarial fixture groups before selecting
  worker count or code generation;
- do not add an unreviewed timeout to static parser patterns, because it changes
  `/parse` output.

The safest strict-parity configuration is an explicitly unbounded regexp2 stack
for legacy static patterns and timed caller patterns, matching the old lack of a
stack-specific failure. That retains a memory-exhaustion risk. If the team
instead chooses the v2 stack bound, record it as an intentional contract
deviation, add an exact error fixture, and call it out in release notes. A
request/body/title length limit or static-pattern timeout is desirable
hardening, but it is a separate API behavior decision and should not be smuggled
into a parity claim.

Do not introduce `regexp2cg` during the initial port. It adds generation,
generated-source review, and potential semantic drift. Only consider it after
benchmarks identify regex execution as a material bottleneck, and then run the
same parity corpus against interpreted and generated engines.

## Wire Contract and Determinism

Define explicit JSON tags on every contract field. Avoid `omitempty`: existing
responses include zero, empty-array, and null fields. Constructors should
initialize every collection so successful responses encode `movieTitles`,
`languages`, `episodeNumbers`, and `absoluteEpisodeNumbers` as `[]`, never
`null`. This is a known Go JSON edge case and is specifically relevant to the
current consumer.

Do not marshal Go enum integers. Keep domain constants typed, but convert them
explicitly to the exact wire strings (`Unknown`, `WebDL`, `BRDisk`,
`PortugueseBR`, `SingleEpisode`, and so on). Add exhaustive tests so a new enum
value cannot silently encode as an empty string.

Capture these endpoint behaviors from the .NET oracle rather than guessing:

- missing body, malformed JSON, JSON `null`, wrong JSON types, unknown
  properties, and content type;
- whitespace-only `title`/`text` and invalid/missing/case-variant `type`;
- null, empty, duplicate, and very large `patterns`/`texts` arrays;
- exact status, content type, and `{ "error": ... }` payload;
- health status string and version source;
- method/path mismatch behavior.

For semantic golden comparisons, decode JSON and compare typed values so
object-key order is not a false failure. Separately keep a few byte-level
contract snapshots for null/empty distinctions, field names, status codes, and
headers. Go's JSON encoder sorts string map keys today, but neither clients nor
tests should make object member order a contract.

The current dictionary responses also define duplicate behavior:

- repeated patterns collapse to one result keyed by the pattern;
- repeated texts in `/match/batch` collapse to one outer result keyed by text;
- the last assignment wins, though identical input makes the value equivalent.

Preserve this behavior even if slices would be a cleaner new API. Test it
explicitly.

### Shared and generated contract data

The highest-value shared artifact is the fixture corpus, not generated
production models. Store each case with a stable ID, request method/path/body,
expected status/headers/body, category tags, and legacy parser version. Both the
.NET oracle generator, Go tests, and Deno consumer contract tests can read the
same JSON/NDJSON data.

Keep the small Go wire structs handwritten initially. Generating them from a new
schema would add a generator/toolchain decision during a behavior-preserving
migration, while OpenAPI cannot express regex execution semantics, stable array
order, or all legacy validation behavior. If the internal parser API later
becomes independently supported, document the already-proven contract in a
checked-in OpenAPI file and generate both Go and TypeScript clients from it in a
follow-up. Do not place these private `/parse` and `/match` routes under the
application's unrelated `/api/v1` OpenAPI merely to satisfy code generation.

## Golden, Differential, and Fixture Strategy

### Corpus ownership

Build the corpus before porting domain logic. The legacy .NET implementation is
the only oracle for initial expected results. A generator should:

1. run a pinned legacy parser build;
2. send canonical HTTP requests rather than calling C# internals, so
   serialization and validation are included;
3. normalize only nondeterministic transport metadata, never response values;
4. write stable case IDs and lexically sorted fixture files;
5. refuse to overwrite existing expected results unless an explicit update flag
   is supplied;
6. record the oracle commit/parser version in a manifest.

Never provide a generic `UPDATE_GOLDEN=1 go test` path that rewrites
expectations from the Go code under test. Golden changes require a readable diff
and a stated behavior reason.

### Fixture taxonomy

Use tagged, reviewable cases rather than one huge opaque snapshot:

- quality: every source/resolution/modifier/revision branch and overlapping
  tokens;
- language: every language, case-sensitive abbreviation, duplicate detection,
  German DL/ML, and stable distinct order;
- release group: anime prefix, exception groups, numeric/hash rejection, suffix
  cleanup;
- movie title: each ordered regex, edition, alternate title, IDs, hashes,
  hardcoded subtitles, reverse-name behavior, and rejected hashes;
- episode: each ordered regex,
  daily/absolute/multi/season/partial/special/mini-series behavior, repeated
  captures, ranges, and rejection;
- common cleanup: extensions, website prefix/postfix, torrent suffix, Unicode
  brackets;
- HTTP: success, validation, malformed input, duplicates, timeouts, and health;
- Unicode: combining marks, non-ASCII case pairs, supplementary code points, and
  titles mixing scripts.

Where possible, add a “near miss” beside each positive fixture. Parser
precedence bugs often pass all happy-path examples while selecting the wrong
earlier regex for an overlapping title.

### Test layers

1. **Pure table tests:** one `_test.go` beside each parser file; assert exact
   typed results and stable ordering.
2. **Regex adapter tests:** constructs, repeated captures, replacement, invalid
   patterns, timeout and stack-limit mapping.
3. **Golden service tests:** load all `testdata/parity` cases and compare typed
   output.
4. **HTTP handler tests:** use `httptest.NewServer`/`httptest.NewRecorder`;
   assert status, headers, exact error JSON, and empty/null fields.
5. **Differential tests:** launch pinned .NET and candidate Go binaries on
   ephemeral ports, replay the corpus, and emit a minimal JSON diff by case ID.
   This is a migration/CI tool, not a permanent production dependency.
6. **Deno consumer integration:** point the real `ParserClient` at the Go server
   and reuse score/entity testing flows that already exercise `/health`,
   `/parse`, and `/match/batch`.
7. **Packaging smoke tests:** run the Docker image as its non-root user and the
   standalone `praxrr-parser[.exe]` next to Praxrr; verify auto-spawn, health
   readiness, termination, and archive layout.

Make tests hermetic: ephemeral ports, `t.TempDir`, no wall-clock sleeps for
readiness, no network after module download, stable fixture sorting, and no
shared mutable parser state. Use `t.Parallel()` only after proving a test does
not mutate the regexp2 timeout clock or global environment.

### Fuzzing

Add native Go fuzz targets after the deterministic corpus is green:

- parsers never panic for arbitrary UTF-8 strings;
- cleanup functions are deterministic and do not create invalid UTF-8;
- valid JSON requests cannot panic handlers;
- match endpoints return within the configured policy for bounded seeds;
- enum/string conversion is total.

Seed fuzzers from representative golden inputs and commit minimized regressions
under `testdata/fuzz`. Do not make an unbounded fuzz campaign a per-PR gate; run
seeds on every `go test`, a short fixed-duration smoke fuzz in scheduled CI, and
longer campaigns manually or nightly.

## Error Handling and Logging

Use three error classes without building a general error framework:

- **Expected no parse:** `(value, false)` or a nil domain result; becomes the
  same zero/null response currently returned.
- **Caller regex failure:** invalid, timeout, or backtracking-limit result; log
  at debug/warn according to current practice and return `false` for that
  pattern.
- **Invariant/startup failure:** static regex cannot compile, server cannot
  bind, or invalid startup configuration; return a contextual error to `main`,
  log once, and exit nonzero.

Internal helpers should return contextual errors such as
`episode regex season-pack: match: ...`. The service boundary may intentionally
contain those errors to preserve the current “parsing failed” behavior, but
tests and structured logs must make the failure diagnosable. Do not reproduce
the bare `catch {}` blocks in `TitleParser.ParseMovieTitle` and
`EpisodeParser.ParseTitle`.

Avoid panic for request data. `MustCompile` is acceptable only for immutable
source constants when the same patterns are compiled in tests; an explicit
startup compilation pass produces better pattern names and failures. Do not
ignore regexp2's returned error, even when examples show `_`.

Keep logs structured and bounded. Preserve source categories (`Parse`, `Match`,
`Health`) but avoid logging complete attacker-controlled regexes or arbitrarily
long release titles at info level. Log stable pattern IDs for built-in regexes
and length/hash plus a safely truncated preview for caller patterns. Maintain
the current optional-service behavior: parser failure should remain visible
while the main application continues to degrade gracefully.

## Concurrency and Resource Bounds

Correctness does not require concurrency in any domain parser. Static compiled
`regexp2.Regexp` values are documented as safe across goroutines, so compile
them once and reuse them. Do not cache unbounded caller patterns globally;
`/match/batch` already amortizes compilation within one request.

For `/match/batch`, start with a bounded worker pool over texts:

```go
workers := min(len(texts), max(1, runtime.GOMAXPROCS(0)))
```

Each worker owns its inner result map. A single collector owns the outer map,
avoiding locks and race risk. Preserve duplicate-text collapse explicitly. Do
not launch one goroutine per text or per text-pattern pair; request arrays are
currently unbounded and regexp2 backtracking is CPU intensive. Benchmark
sequential versus bounded execution before retaining concurrency at all.

The HTTP server should set `ReadHeaderTimeout`, `ReadTimeout`, `WriteTimeout`,
and `IdleTimeout`, and perform graceful shutdown from signal context. However,
request cancellation does not replace regexp2's match timeout: a backtracking
matcher must reach its own cancellation check. Every goroutine must have an
obvious owner and exit condition.

Request-size and collection-count limits are strongly recommended against
CPU/memory abuse, but they change previously accepted inputs. Measure production
request sizes, capture the legacy response for oversized cases, and approve
limits as an explicit cutover decision rather than silently adding them.

## Dependency and Toolchain Pinning

Prefer `github.com/dlclark/regexp2/v2` over the legacy v1 module because v2 is
the maintained API line, documents its memory/backtracking behavior, and
provides explicit safety controls. Pin an exact tagged release in `go.mod`; as
of this research, available v2 tags extend through `v2.3.0`. Never use `@latest`
in committed build or CI commands.

The repository should declare the toolchain in all execution environments:

- `go.mod`: minimum compatible language/toolchain requirement (at least Go 1.25
  for regexp2/v2);
- `toolchain` directive: reviewed patch release used by contributors;
- `mise.toml`: same Go patch for local development;
- GitHub Actions: the same patch via `actions/setup-go` with dependency caching;
- `Dockerfile.parser`: matching builder image, preferably pinned by digest for
  release builds.

Go 1.26.5 is the current supported patch release at the time of research, while
regexp2/v2 requires only Go 1.25. A reasonable policy is `go 1.25.0` plus
`toolchain go1.26.5` if the code intentionally avoids 1.26-only language
features; using `go 1.26.0` is simpler if the project does not promise older
builder compatibility. Decide once and keep every surface aligned.

Commit `go.sum`. Run `go mod tidy` and fail CI if it produces a diff. Do not
vendor a single small dependency unless offline/reproducibility policy requires
all dependencies to be vendored. Add `govulncheck ./...`; pin any nonstandard Go
tool through the module's `tool` directive or an exact version, rather than
installing `@latest` in CI.

## CI and Quality Gates

Add a focused parser job triggered by parser, fixture, Docker, task, and
release-workflow changes. The minimum PR gate should run from
`packages/praxrr-parser`:

```text
gofmt check
go vet ./...
go test ./...
go test -race ./...
govulncheck ./...
go mod tidy + clean-diff check
```

Also run:

- golden parity on every parser change;
- differential .NET-versus-Go parity until legacy retirement;
- cross-build `GOOS/GOARCH go build` for every release matrix target;
- Docker build plus `/health`, `/parse`, and `/match` smoke checks;
- standalone packaging smoke on Linux and Windows (at minimum build; execute
  where runners permit);
- existing Deno compatibility and affected route tests;
- `deno task check:dist-paths` after changing build output paths.

`go test -race` executes code paths; it does not prove unexercised concurrency
safe. Include a batch match test with multiple workers and repeated shared
compiled expressions. Run parser benchmarks and short fuzzing on a scheduled
workflow rather than making noisy performance thresholds a PR gate.

Wire root-level developer tasks such as `deno task test:parser`, `check:parser`,
and `build:parser` so contributors do not need tribal commands. Update
`scripts/test.ts` aliases only if the root test orchestrator is intended to
include Go; avoid hiding failures behind a shell pipeline.

## Concrete Repository Patterns to Preserve

- **Optional-service isolation:** the application already returns `null`/partial
  behavior when parser health fails. Preserve that boundary during rollout.
- **Real consumer contract tests:** route tests create an HTTP parser stub and
  exercise cache/version behavior through the actual client. Add Go-server
  variants rather than replacing them with mocks.
- **Explicit artifact naming:** release archives and `spawn.ts` agree on
  `praxrr-parser` and `praxrr-parser.exe`. Keep the names stable.
- **Non-root containers and health dependency:** the parser image runs as an
  unprivileged user and compose waits for health. Preserve both.
- **Path-scoped CI:** compatibility workflows already include
  `packages/praxrr-parser/**` and parser Docker/task surfaces. Extend, do not
  create disconnected gating rules.
- **Repository-root `dist/`:** all Go outputs must land below root `dist/`,
  matching the stated build artifact policy.
- **Stable version cache namespace:** the parser health version invalidates
  parsed-result caches. Inject the application release/parser behavior version
  at link time and test it; do not leave a hard-coded `1.0.0` forever.

## Existing Anti-Patterns Not to Carry Forward

1. **No parser-native tests.** The C# project contains production source but no
   test project or fixtures. Do not begin porting before freezing observable
   outputs.
2. **Swallowed exceptions.** Title and episode parsing catch every exception and
   discard it. Preserve public fallback behavior, not internal invisibility.
3. **Contract duplication without an executable check.** C# records and
   TypeScript interfaces mirror each other manually. Shared golden HTTP cases
   should become the executable contract.
4. **Implementation-shaped folders.** C# `Models`, `Parsers/Common`, and static
   classes should not be mechanically reproduced as Go `types`, `common`, or one
   package per class.
5. **Implicit ordering through dictionaries/maps.** Language and regex result
   order must not depend on Go map iteration. Sort fixture discovery and use
   slices for ordered domain output.
6. **Unbounded batch work.** The current `Parallel.ForEach` has no request-level
   worker/size policy. Do not replace it with even less bounded goroutine
   fan-out.
7. **Builds mutate tracked source.** Current standalone tasks remove
   `Directory.Build.props` and run `git checkout` to restore it. Go builds
   should be pure commands that never alter tracked files.
8. **Runtime-specific spawn configuration.** `spawn.ts` sets `ASPNETCORE_URLS`
   and `ASPNETCORE_ENVIRONMENT`. Replace these with a documented Go flag/env
   contract while preserving host/port selection and shutdown.
9. **Hard-coded fallback version.** `Program.cs` defaults to `1.0.0`, yet the
   application uses version for cache invalidation. Make build-version injection
   explicit and test fallback behavior for local builds.
10. **Formatter scaffolding without enforcement.** `scripts/go-tools.sh` exists,
    but Go is absent from `mise.toml`, root tasks, and CI. Use standard
    `gofmt`/`go vet` first; add a pinned broader linter only if its ownership is
    clear.

## Migration Sequencing

### Phase 0 — Freeze behavior

- Inventory every endpoint, regex, option, capture consumed, enum string,
  default field, deployment surface, and build target.
- Build the .NET-owned fixture corpus and HTTP oracle runner.
- Add legacy contract tests for malformed input, duplicates, Unicode, and
  timeouts.
- Establish fixture review/update policy and record the oracle version.

Exit gate: rerunning the oracle produces a byte-identical corpus and all current
consumer tests pass.

### Phase 1 — Foundation (#2)

- Add the pinned Go module/toolchain and the minimal package layout.
- Implement explicit contract structs, enum conversion, regex adapter, cleanup
  helpers, golden loader, and compile-all-pattern tests.
- Add root tasks and a non-blocking focused CI job, then make it required once
  stable.

Exit gate: `go test`, race, vet, vulnerability, static regex compilation, and
foundation goldens pass.

### Phase 2A — Leaf domain parsers (#3)

- Port quality, language, and release group first; they have simpler outputs and
  exercise the regex adapter thoroughly.
- Port movie title and episode parsers literally, preserving regex order and
  capture behavior.
- For every discrepancy, first minimize it into a fixture; change the adapter or
  port only with oracle evidence.

Exit gate: every domain fixture matches and no known case is skipped or marked
“close enough.”

### Phase 2B — Orchestration and HTTP (#4)

- Compose parse results without HTTP knowledge.
- Add the four exact routes, validation, health/version, logging, bounded batch
  execution, server timeouts, and graceful shutdown.
- Run typed and selected byte-level differential HTTP tests against .NET and Go.

Exit gate: status, headers, JSON values, null/empty behavior, duplicate
behavior, invalid regexes, and timeout classification match the oracle.

### Phase 3 — Integration and cutover (#5)

- Teach `spawn.ts` the Go bind/version/log contract while keeping executable
  names stable.
- Replace Docker stages, compose watches, development tasks, standalone builds,
  release matrix, docs, and prerequisites.
- Run Go as a temporary alternate service in CI; do not make production requests
  depend on two parsers.
- Verify Docker non-root health, Linux/Windows/macOS cross-builds, archive
  layout, app consumer flows, and E2E tests.
- Bump parser behavior version so existing caches cannot conceal cutover
  differences.

Exit gate: all release surfaces build from Go, real app flows use Go, and CI has
no .NET-only parser dependency except the temporary differential oracle.

### Phase 4 — Retirement

- Remove C# source, `Parser.csproj`, `Directory.Build.props`, appsettings, .NET
  images/toolchain setup, `ASPNETCORE_*` wiring, and documentation references in
  one auditable cleanup.
- Keep the immutable legacy golden corpus and oracle version metadata.
- Remove differential execution only after the final full parity report is
  attached to the PR; keep ordinary golden tests permanently.

Exit gate: repository search finds no active .NET parser build/runtime
dependency, all issue #1–#5 acceptance criteria are evidenced, and
release/standalone/Docker CI is green.

## Maintainability Rules After Cutover

- Every parser behavior change adds positive and near-miss fixtures before
  implementation.
- Regex definitions carry stable IDs and a short purpose comment; regex order
  changes require a fixture diff.
- Parser version changes whenever observable parse/match semantics change, not
  merely on binary rebuilds.
- Dependency/toolchain upgrades run the complete golden, Unicode, timeout, race,
  benchmark-smoke, and Docker suites.
- Generated goldens are never reformatted or updated in bulk without an oracle
  version and reviewable change report.
- Performance optimizations require representative benchmarks and parity proof;
  no regex rewrite is accepted solely because it appears equivalent.
- Keep the internal surface small. Export only what the executable/handler
  actually consumes.

## Open Questions

1. **Which exact Go and regexp2 versions will be the support baseline?**
   Recommend regexp2/v2 at an exact tag and a repository-wide Go 1.26.5
   toolchain, with the `go` directive set to 1.25 only if older builder
   compatibility is deliberate.
2. **Will strict parity explicitly disable regexp2/v2's default backtracking
   stack bound?** That is closest to existing semantics but preserves a
   memory-exhaustion risk. A bound is safer but must be documented as a behavior
   difference.
3. **What timing tolerance defines 100 ms timeout parity?** regexp2's timeout
   clock is approximate and process-wide. Define classification and an
   acceptable upper bound rather than asserting an exact wall-clock millisecond.
4. **Are request body/title/pattern/text count limits authorized in this
   migration?** Recommend a separate hardening decision unless current clients
   and the legacy oracle prove a compatible limit.
5. **What is the canonical parser behavior version?** Recommend injecting the
   app release version plus an explicit parser contract revision, and bumping it
   for semantic changes so cache invalidation is reliable.
6. **Where will the legacy oracle run after source retirement?** Recommend
   retaining a pinned final binary/container only as a CI migration artifact
   until merge, then preserving its generated corpus and manifest rather than
   shipping or maintaining two runtimes.
7. **Must unsupported/malformed JSON behavior match ASP.NET byte-for-byte or
   only status/schema?** The answer determines how much framework-generated
   error payload must be captured and reproduced.
8. **Should the internal parser API receive a checked-in OpenAPI document now?**
   Recommend no codegen in the parity port; use the shared executable fixture
   contract first and document/generate later if the service becomes
   independently supported.
9. **Is batch output ordering observed anywhere outside JSON object decoding?**
   Current TypeScript uses `Object.entries`, so semantic key/value parity should
   suffice, but confirm no snapshot/log consumer compares raw JSON bytes.
10. **Can static `/parse` regexes remain unbounded?** Strict parity says yes;
    production security may justify a follow-up timeout/size policy after
    adversarial benchmarks.
