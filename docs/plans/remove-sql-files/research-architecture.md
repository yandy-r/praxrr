# Architecture Research: remove-sql-files

## System Overview

The PCD ingestion pipeline loads configuration data through a four-layer system (schema, base,
tweaks, user) that compiles SQL operations into an in-memory SQLite cache. A migration from
SQL-file-based entity definitions to YAML-based "portable entities" has been completed, but the
legacy SQL ingestion paths remain as a backwards-compatible fallback controlled by the
`PRAXRR_PCD_MIGRATION_MODE` environment variable (defaults to `hybrid`). Removing SQL files means
eliminating the `sql-only` mode, the SQL file reading in `importBaseOps`, the
`loadOperationsFromDir` function's `.sql` filter for schema/tweaks layers, and the SQL ops files
from `packages/praxrr-schema/ops/` and `packages/praxrr-db/ops/`.

## SQL File Inventory

### PCD/Schema SQL Files (REMOVAL TARGETS)

#### Schema Package (3 files - loaded at compile time as schema layer)

- `/packages/praxrr-schema/ops/0.schema.sql`: PCD schema DDL (CREATE TABLE statements for all PCD
  tables). ~26KB. This is the foundational schema used by `generate-pcd-types.ts` and executed as
  the first layer during cache build.
- `/packages/praxrr-schema/ops/1.languages.sql`: Seed INSERT statements for the `languages` table.
  ~900 bytes.
- `/packages/praxrr-schema/ops/2.qualities.sql`: Seed INSERT statements for the `qualities` and
  `quality_api_mappings` tables. ~3.3KB.

#### DB Package (53+ SQL ops files - loaded as base layer)

- `/packages/praxrr-db/ops/0.rosettarr.sql`: Initial base entity import (~1.4MB). The original bulk
  SQL export of all PCD entities.
- `/packages/praxrr-db/ops/1.*.sql` through `/packages/praxrr-db/ops/58.*.sql`: Incremental base ops
  (updates, fixes, new entities). These range from ~300 bytes to ~147KB.

#### Runtime Data Copies (auto-cloned, gitignored)

- `/config/data/databases/<uuid>/deps/praxrr-schema/ops/*.sql`: Cloned schema dependency.
- `/config/data/databases/<uuid>/ops/*.sql`: Cloned base ops.
- `/dist/dev/data/databases/<uuid>/deps/praxrr-schema/ops/*.sql`: Dev build copies.
- `/dist/dev/data/databases/<uuid>/ops/*.sql`: Dev build copies.

## Relevant Components

### Core Ingestion Pipeline

- `/packages/praxrr-app/src/lib/server/pcd/core/manager.ts`: PCDManager singleton. Orchestrates
  link/sync/initialize lifecycle. Contains `importBaseOpsWithOrchestration()` (lines 488-518) which
  implements the `sql-only`/`hybrid` mode switching and legacy fallback logic.
- `/packages/praxrr-app/src/lib/server/pcd/ops/importBaseOps.ts`: Primary SQL file ingestion. Reads
  `.sql` files from `ops/` directory, parses metadata headers, computes content hashes, stores into
  `pcd_ops` table. Contains the hybrid/sql-only branching logic and migration entity suppression.
- `/packages/praxrr-app/src/lib/server/pcd/ops/loadOps.ts`: Loads all operations for cache build.
  Calls `loadOperationsFromDir()` for schema layer (from `deps/<schema>/ops/`) and tweaks layer.
  Base/user ops load from DB.
- `/packages/praxrr-app/src/lib/server/pcd/database/cache.ts`: PCDCache class. Builds in-memory
  SQLite by executing all operations from `loadAllOperations()`. Does not directly interact with SQL
  files -- receives `Operation` objects.
- `/packages/praxrr-app/src/lib/server/pcd/database/compiler.ts`: Orchestrates cache build. Calls
  `cache.build()` which calls `loadAllOperations()`.
- `/packages/praxrr-app/src/lib/server/pcd/utils/operations.ts`: `loadOperationsFromDir()` function
  that reads `.sql` files from a directory. Used for schema layer and tweaks layer. Also contains
  the dead-code `loadAllOperations()` function (shadowed by the one in `loadOps.ts`).

### Migration System (YAML path)

- `/packages/praxrr-app/src/lib/server/pcd/migration/reader.ts`: Reads YAML/JSON entity files from
  `entities/` directories. Returns `MigrationEntityCandidate` objects with deserializer functions.
  The YAML path that replaces SQL ingestion.
- `/packages/praxrr-app/src/lib/server/pcd/migration/converter.ts`: Converts PCD cache data to
  portable YAML format. Used for entity export.
- `/packages/praxrr-app/src/lib/server/pcd/migration/enumerateEntities.ts`: Enumerates all entity
  types/names from cache for migration.
- `/packages/praxrr-app/src/lib/server/pcd/migration/yamlFormatter.ts`: Deterministic YAML output
  formatter.
- `/packages/praxrr-app/src/lib/server/pcd/migration/migrationImportUtils.ts`: Stable identity types
  and import ordering for migration entities.
- `/packages/praxrr-app/src/lib/server/pcd/migration/parityVerifier.ts`: Verifies SQL-only and
  entity-based ingestion produce identical results. References `sql-only` mode explicitly (line
  389).

### Entity Serialization/Deserialization

- `/packages/praxrr-app/src/lib/server/pcd/entities/serialize.ts`: Serializes PCD cache entities to
  portable format (for YAML export).
- `/packages/praxrr-app/src/lib/server/pcd/entities/deserialize.ts`: Deserializes portable entities
  back into PCD ops (for YAML import).
- `/packages/praxrr-app/src/lib/server/pcd/entities/validate.ts`: Validates portable entity data
  before deserialization.

### Export Pipeline

- `/packages/praxrr-app/src/lib/server/pcd/ops/exporter.ts`: Exports draft ops as `.sql` files to
  the PCD repo. Generates `{number}.{slug}.sql` filenames (line 373). This writes SQL ops to the
  repo for git commit/push.
- `/packages/praxrr-app/src/lib/server/pcd/ops/writer.ts`: Writes operations to `pcd_ops` DB table.
  The repo import context generates `.sql`-suffixed filenames (line 68).
- `/packages/praxrr-app/src/lib/server/pcd/utils/git.ts`: `getMaxOpNumber()` scans committed
  `ops/*.sql` files to determine next op number for export.

### Configuration

- `/packages/praxrr-app/src/lib/server/utils/config/config.ts`: Defines `PCDMigrationIngestionMode`
  type (`'sql-only' | 'hybrid'`), `pcdMigrationIngestionMode` and `pcdMigrationAllowLegacyFallback`
  config properties. Parses `PRAXRR_PCD_MIGRATION_MODE` env var.

### Code Generation

- `/scripts/generate-pcd-types.ts`: Generates TypeScript types from
  `packages/praxrr-schema/ops/0.schema.sql`. Reads the schema SQL, executes it in an in-memory
  SQLite DB, introspects tables, generates types. This is a build-time tool that directly depends on
  the schema SQL file.

### Validation Scripts (in schema package)

- `/packages/praxrr-schema/scripts/validateLanguages.sh`: Validates languages against Radarr/Sonarr
  source. Reads from `ops/1.languages.sql`.
- `/packages/praxrr-schema/scripts/validateQualities.sh`: Validates qualities against Radarr/Sonarr
  source. Reads from `ops/2.qualities.sql`.
- `/packages/praxrr-schema/scripts/generate-schema-diagram.sh`: Generates ER diagram from

### PCD Dependency System

- `/packages/praxrr-app/src/lib/server/pcd/git/dependencies.ts`: Clones schema dependency repos.
  After clone, keeps only `.git`, `ops/`, and `pcd.json`. The `ops/` directory assumption is
  hardcoded.

## Data Flow

### Current Flow (Hybrid Mode - Default)

1. **Repository Clone/Sync**: PCD repo is cloned locally. Dependencies (schema) are cloned into
   `deps/` subdirectory.
2. **Dependency Resolution**: `processDependencies()` clones `praxrr-schema` repo, checks out the
   version tag, keeps only `.git`, `ops/`, `pcd.json`.
3. **Base Op Import** (`importBaseOps()`):
   - Reads SQL files from `<pcd>/ops/*.sql` directory.
   - In hybrid mode: also calls `readMigrationEntitySources()` to read YAML files from
     `<pcd>/entities/`.
   - SQL entries with matching stable identities in migration entities are suppressed (migration
     wins).
   - Remaining SQL entries and migration entities are stored in `pcd_ops` table.
4. **Cache Build** (`loadAllOperations()` then `cache.build()`):
   - **Schema layer**: Reads `.sql` files from `deps/<schema>/ops/` via `loadOperationsFromDir()`.
   - **Base layer**: Reads from `pcd_ops` DB table (published, then draft).
   - **Tweaks layer**: Reads `.sql` files from `<pcd>/tweaks/` via `loadOperationsFromDir()`.
   - **User layer**: Reads from `pcd_ops` DB table.
   - All operations are executed sequentially in the in-memory SQLite DB.

### Schema Layer Loading Detail

`loadOps.ts:resolveSchemaOpsPath()` dynamically finds the schema dependency directory by scanning
`deps/` for a directory containing "schema" in the name, then appends `/ops`. The
`loadOperationsFromDir()` function reads all `.sql` files from that path.

### Export Flow (write path - creates .sql files)

Draft ops in `pcd_ops` are bundled into a single `.sql` file with metadata headers, committed and
pushed to the PCD repo. This uses `getMaxOpNumber()` to scan existing `ops/*.sql` committed files.

## SQL vs YAML Loading Paths

### SQL-Only Path (Legacy)

Controlled by `PRAXRR_PCD_MIGRATION_MODE=sql-only`:

- `importBaseOps()` reads only `<pcd>/ops/*.sql` files.
- Each SQL file is parsed for metadata headers (`-- @operation:`, `-- @entity:`, `-- @name:`).
- Content is hashed and stored in `pcd_ops` table.
- No migration entity reading occurs.

### Hybrid Path (Current Default)

Controlled by `PRAXRR_PCD_MIGRATION_MODE=hybrid` (or unset):

- `importBaseOps()` reads both SQL files from `<pcd>/ops/` and YAML entities from `<pcd>/entities/`.
- Stable identity conflict detection ensures no duplicate definitions.
- Migration entities (YAML) take precedence over SQL entries with matching stable identities.
- When `pcdMigrationAllowLegacyFallback=true` and hybrid ingestion fails with a
  `MigrationReaderError`, falls back to `sql-only` mode.
- When `pcdMigrationAllowLegacyFallback=false` (default), hybrid failures are fatal.

### Fallback Mechanism in Manager

`PCDManager.importBaseOpsWithOrchestration()` (lines 488-518 of manager.ts):

1. If mode is `sql-only`: directly calls `importBaseOps` with `sql-only`.
2. If mode is `hybrid`: tries hybrid import. On `MigrationReaderError` and
   `pcdMigrationAllowLegacyFallback=true`, retries with `sql-only`.
3. Non-`MigrationReaderError` errors always propagate.

### Schema Layer (Always SQL, No YAML Equivalent Yet)

The schema layer (`0.schema.sql`, `1.languages.sql`, `2.qualities.sql`) is always loaded from `.sql`
files via `loadOperationsFromDir()`. There is no YAML representation of the schema DDL. This is the
most critical SQL dependency because:

- It defines the in-memory SQLite table structure.
- `generate-pcd-types.ts` depends on it for TypeScript type generation.
- Validation scripts in the schema package parse these SQL files.

## Integration Points

### Files That Must Change

1. **`importBaseOps.ts`**: Remove SQL file reading loop, `parseMetadata()`, SQL stable identity
   derivation, SQL/migration overlap suppression logic, `sql-only` branch,
   `UNPREFIXED_SEQUENCE_BASE` constant. In YAML-only mode, only migration entity deserialization
   remains.
2. **`loadOps.ts`**: `loadOperationsFromDir()` call for schema layer must be replaced or the schema
   itself must be provided differently (since `0.schema.sql` creates the PCD tables). Tweaks layer
   also uses `loadOperationsFromDir()`.
3. **`utils/operations.ts`**: `loadOperationsFromDir()` function and the dead `loadAllOperations()`
   function. Both filter for `.sql` files only. These need to either be removed or adapted if tweaks
   layer transitions to YAML.
4. **`core/manager.ts`**: Remove `importBaseOpsWithOrchestration()` fallback logic. Remove
   `sql-only` code path. Simplify to always use the migration entity path.
5. **`config/config.ts`**: Remove `PCDMigrationIngestionMode` type, `pcdMigrationIngestionMode`,
   `pcdMigrationAllowLegacyFallback`, `parsePCDMigrationMode()`. Or simplify to just `yaml-only`.

7. **`utils/git.ts`**: `getMaxOpNumber()` scans `ops/*.sql` for export numbering. May need updating

8. **`ops/exporter.ts`**: Generates `.sql` files for export. The export pipeline writes SQL ops to

9. **`ops/writer.ts`**: Generates `.sql`-suffixed filenames for repo import context (line 68).

## Key Dependencies

- **`@jsr/db__sqlite`**: SQLite library used for in-memory PCD cache. The schema SQL must be
  executed to create tables regardless of data format.
- **`@std/yaml`**: YAML parser used by the migration reader for entity files.
- **Kysely**: SQL query builder. Used for type-safe queries and compiled query generation (write
  path).
- **`pcd_ops` table (app DB)**: Stores imported base/user ops as SQL text. Migration entities are
  deserialized into SQL ops stored here.

## Architectural Patterns

- **DB-First Pattern**: Base and user ops are stored in `pcd_ops` table, not as files. SQL files are
  imported into the DB on link/sync. The cache build reads from DB for base/user layers and from
  files only for schema/tweaks layers.
- **Layer Ordering**: Schema (files) -> Base (DB) -> Tweaks (files) -> User (DB). Schema and tweaks
  are the only layers still reading SQL files at compile time.
- **Stable Identity**: Entities are identified by a stable key (e.g., `custom_format_name=<name>`)
  for deduplication between SQL and YAML sources.
- **Hybrid Suppression**: In hybrid mode, migration YAML entities suppress overlapping SQL ops by
  stable identity. This is the mechanism that makes SQL files redundant when YAML equivalents exist.
- **Export as SQL**: The exporter writes draft ops to repos as `.sql` files. This is separate from
  ingestion and may need its own migration plan.

## Edgecases

- The schema DDL (`0.schema.sql`) has no YAML equivalent and is structurally different from entity
  data -- it contains CREATE TABLE statements, not INSERT/UPDATE data. Removing this requires a
  separate approach (perhaps keeping it as SQL or converting to a different schema definition
  format).
- The `generate-pcd-types.ts` script directly reads and executes `0.schema.sql` to introspect the
  database structure. Any schema format change must preserve this code generation capability.
- The tweaks layer reads `.sql` files from `<pcd>/tweaks/` directory. If this layer has active
  usage, it needs a YAML migration too, or the `.sql` filter in `loadOperationsFromDir` must be
  preserved for tweaks.
- The exporter writes SQL files to repos. If the goal is to eliminate SQL from the ingestion side
  only, the export path can remain as-is. But if repos should stop containing `.sql` ops, the export
  format must also change.
- `getMaxOpNumber()` scans committed `ops/*.sql` files for op numbering. If SQL ops files are
  removed from repos, this function breaks for existing repos with committed SQL history.
- The `packages/praxrr-db/entities/` directory already contains YAML files for all entity types
  (custom-formats, delay-profiles, media-management, metadata-profiles, quality-profiles,
  regular-expressions). These are the YAML replacements for the SQL ops.
- Runtime data copies under `config/data/` and `dist/dev/data/` are clones of the repo -- they will
  stop containing SQL files automatically when the repo packages stop containing them.
- The `docs/pcdReference/` SQL files are documentation/reference copies and are not loaded by any
  runtime code.
