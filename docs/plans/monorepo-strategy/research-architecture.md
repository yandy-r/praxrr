# Architecture Research: monorepo-strategy

## System Overview

The repo centers on a SvelteKit/Deno 2.x application rooted in `/src`, where `hooks.server.ts` wires configuration, the SQLite `praxrr.db`, migrations, the PCD manager, and the job queue before any route resolves. Business logic lives in `packages/praxrr-app/src/lib/server` (PCD lifecycle, sync scheduling, Arr client helpers, notifications) while `packages/praxrr-app/src/lib/client`/`packages/praxrr-app/src/routes` deliver the UI and API surface that consume compiled PCD cache state. The monorepo strategy (see `docs/plans/monorepo-strategy/feature-spec.md`) layers Deno workspace members under `/packages` (`praxrr-api` currently, with planned `praxrr-db` and `praxrr-schema`), shared scripts, and CI gates so schema, DB ops, and the app can evolve together without cross-repo coordination.

## Relevant Components

- `/src/`: SvelteKit entry point where `hooks.server.ts` bootstraps config, DB, PCD caches, jobs, and auth; `packages/praxrr-app/src/routes/**` hosts the UI pages (`+page.svelte/+page.server.ts`) plus API handlers (`packages/praxrr-app/src/routes/api/v1/**`) that drive the monorepo strategy features.
- `/packages/praxrr-app/src/lib/server/pcd/`: PCD manager (`core/manager.ts`), Git dependency helpers, manifest validation, operation loaders (`ops/loadOps.ts`), and the cache/compiler (`database/`) that replay schema → base → tweaks → user ops and feed the rest of the stack.
- `/packages/praxrr-app/src/lib/server/db/`: Database singleton (`db.ts`), migrations runner (`migrations.ts` + `/migrations/*.ts`), and query layers (`queries/**` such as `arrInstances.ts`, `pcdOps.ts`, `jobQueue.ts`) that persist PCD operations, Arr instance metadata, sync state, and job scheduling data the monorepo strategy must keep compatible with the `packages/praxrr-db` workloads.
- `/packages/praxrr-app/src/lib/server/sync/` + `/packages/praxrr-app/src/lib/server/jobs/`: Sync processor (`sync/processor.ts`), section registry, Arr client factory, job dispatcher/scheduler (`jobs/schedule.ts`, `jobs/dispatcher.ts`), and cleanup helpers that push validated PCD cache data to Arr instances and run background maintenance; these services rely on the compiled data the monorepo strategy keeps consistent across packages.
- `/packages`: `packages/praxrr-api` hosts the OpenAPI bundle (`mod.ts`, `openapi.json`) and types consumed by both client and server; the monorepo strategy adds `packages/praxrr-db` and `packages/praxrr-schema` workspace members (per `docs/plans/monorepo-strategy/feature-spec.md`) so database ops and schema DDL live alongside the app and expose `ops/`, `pcd.json`, and new `deno.json` configs.
- `/scripts`: Shared tooling such as `scripts/dev.ts` (runs parser + Vite + optional Arr containers), `scripts/generate-pcd-types.ts` (reads `packages/praxrr-schema/ops/0.schema.sql` and outputs `packages/praxrr-app/src/lib/api/v1.d.ts`), and `scripts/bundle-api.ts` plus `deno task publish:api` that package the API spec; these scripts are the natural integration surface for monorepo tasks that stitch packages together.
- `/docs/api/v1/` & `/docs/ARCHITECTURE.md`: API contracts (`openapi.yaml`, `schemas/`, `paths/`) and the high-level architecture reference used to reason about how PCD layers, routes, and CI requirements interact as the monorepo strategy grows.

## Data Flow

On startup `hooks.server.ts` sequences `config.init()`, `db.initialize()`, `runMigrations()`, `pcdManager.initialize()`, `initializeJobs()`, and the parser/service bindings so every route executes with an in-memory PCD cache and a healthy job queue. API routes under `packages/praxrr-app/src/routes/api/v1/**` (PCD import/export, Arr status, cleanup) plus UI `load` functions under `packages/praxrr-app/src/routes/media-management`, `packages/praxrr-app/src/routes/arr`, etc., read from shared stores that ultimately call `packages/praxrr-app/src/lib/server/pcd` cache helpers, `db/queries`, and `sync` registries. Writes hit the same layers: user actions create `pcd_ops` rows via writer modules, which trigger re-compilation of the cache (schema/base/tweaks/user order) before the sync processor (`packages/praxrr-app/src/lib/server/sync/processor.ts`) marks sections pending, schedules jobs (`jobs/schedule.ts`), and dispatches them to Arr clients (`$arr/factory.ts`). Background tasks (`jobs/dispatcher.ts`) also call `pcdManager` for linking/syncing remote databases, so new monorepo strategy resources (schema DDL under `packages/praxrr-schema`) must stay accessible to the PCD loader and `scripts/generate-pcd-types.ts` that derive shared typings.

## Integration Points

New monorepo-strategy code touches the Deno workspace manifest `deno.json` (workspace members + shared import aliases) and the GitHub workflows under `.github/workflows/` (`ci.yml`, new `compatibility.yml`, `publish-db.yml`, `publish-schema.yml`) described in `docs/plans/monorepo-strategy/feature-spec.md`. Schema/data assets go under `/packages/praxrr-schema/ops` and `/packages/praxrr-db/ops`, which feed `packages/praxrr-app/src/lib/server/db` migrations, `packages/praxrr-app/src/lib/server/pcd/ops/loadOps.ts`, and `scripts/generate-pcd-types.ts`. API contract changes live in `/docs/api/v1/**` and are bundled via `/scripts/bundle-api.ts`/`packages/praxrr-api`; those artifacts are consumed by the UI stores/routes and by release automation. Runtime glue such as `hooks.server.ts`, `packages/praxrr-app/src/lib/server/pcd/core/manager.ts`, `packages/praxrr-app/src/lib/server/sync/processor.ts`, and job schedulers in `/packages/praxrr-app/src/lib/server/jobs/` are the mount points for features that need to stay consistent across the new packages, while the `packages/praxrr-app/src/lib/client/*` and `packages/praxrr-app/src/routes/**` layers remain the outward API for monorepo-driven capabilities.

## Key Dependencies

- Deno 2.x runtime with workspace support plus the `deno.json` task suite (dev/build/check/test) that glues `/src`, `/packages`, and `/scripts` together.
- SvelteKit (with `npm:vite`, Vite dev server, `svelte-check`, Tailwind) for the UI routes and client stores under `/packages/praxrr-app/src/lib/client` and `/packages/praxrr-app/src/routes`.
- SQLite and Kysely via `@soapbox/kysely-deno-sqlite` powering `packages/praxrr-app/src/lib/server/db/db.ts`, migrations, queries, and the PCD cache compilation system.
- Sync/job tooling: `croner` for schedules, Arr client helpers (`$arr/factory.ts`, `$arr/types.ts`), and the `.NET` parser service (`packages/praxrr-parser`) used by test flows.
- Git + related automation (`git subtree split`, `dorny/paths-filter`, `release-please`) described in `docs/plans/monorepo-strategy/feature-spec.md` for extracting/mirroring `/packages/praxrr-db` and `/packages/praxrr-schema` while keeping compatibility CI gating.
- Node/npm modules (`marked`, `simple-icons`, `highlight.js`, `openapi-typescript`, etc.) that feed the Vite build, the generated API types (`packages/praxrr-app/src/lib/api/v1.d.ts`), and the `packages/praxrr-api` bundle consumed by both client and release artifacts.
