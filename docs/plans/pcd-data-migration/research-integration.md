# Integration Research: pcd-data-migration

## API Endpoints

### Existing Related Endpoints

- `GET /api/v1/pcd/export?databaseId=&entityType=&name=` — queries the in-memory cache for the
  requested entity and returns the portable JSON/YAML data generated by the serialization layer
  (`packages/praxrr-app/src/routes/api/v1/pcd/export/+server.ts:17`). This matches Phase 2’s “JSON
  Exchange + TRaSH Adapter” focus from the research report (`research/data-schema/report.md:102`),
  which emphasizes publishing structured payloads without changing the SQL runtime.
- `POST /api/v1/pcd/import` — validates the body for `databaseId`, `layer`, `entityType`, and
  portable payload, runs the portable schema checks, and calls the deserialization helpers to write
  SQL operations (`packages/praxrr-app/src/routes/api/v1/pcd/import/+server.ts:30`). The handler
  enforces layer permissions with `canWriteToBase`, keeping value-guard behavior locked in place as
  the report stresses—the “value guard prototype” is the go/no-go gate
  (`research/data-schema/report.md:21`, `research/data-schema/report.md:85`).
- `GET /api/v1/pcd/{databaseId}/lidarr-metadata-profiles` — lists Lidarr metadata profiles after
  confirming the schema exists in the cache; it also surfaces the precalculated counts needed by the
  UI
  (`packages/praxrr-app/src/routes/api/v1/pcd/[databaseId]/lidarr-metadata-profiles/+server.ts:279`).
- `POST /api/v1/pcd/{databaseId}/lidarr-metadata-profiles` — validates the payload, enforces layer
  guardrails, and delegates to the metadata profile service to insert operations
  (`packages/praxrr-app/src/routes/api/v1/pcd/[databaseId]/lidarr-metadata-profiles/+server.ts:321`).

### Route Organization

Routes live under `packages/praxrr-app/src/routes/api/v1/`, so every API `+server.ts` is a SvelteKit
handler wired to `/api/v1/<feature>`. The PCD surface has its own subfolder plus dynamic
`[databaseId]` segments and query parameters for import/export
(`packages/praxrr-app/src/routes/api/v1/pcd/export/+server.ts:1`,
`packages/praxrr-app/src/routes/api/v1/pcd/import/+server.ts:1`). Authentication middleware
(`packages/praxrr-app/src/hooks.server.ts:185`) runs before every route, relies on
`getAuthState`/`isPublicPath` (`packages/praxrr-app/src/lib/server/utils/auth/middleware.ts:37`),
and blocks API calls unless the session, API key, or auth bypass is valid. Public endpoints like
`/api/v1/health` are whitelisted in that middleware, so new migration APIs inherit the same
guardrails.

## Database

### Relevant Tables

- `pcd_ops` — stores every base/user SQL operation, captures ordering, metadata, desired state,
  hashes, and foreign keys to `database_instances` plus self-references for superseding operations
  (`packages/praxrr-app/src/lib/server/db/migrations/041_create_pcd_ops.ts:35`).
- `pcd_op_history` — records the result of each operation application (applied/conflicted/dropped)
  along with `conflict_reason` and `batch_id`, supporting diagnostics and the “value guard”
  instrumentation the report calls the critical gate
  (`packages/praxrr-app/src/lib/server/db/migrations/042_create_pcd_op_history.ts:14`;
  `research/data-schema/report.md:21`).
- `database_instances` — tracks linked PCD repositories with sync strategy, auto-pull, enablement,
  and timestamps; every operation references this table so multiple repos can coexist
  (`packages/praxrr-app/src/lib/server/db/migrations/008_create_database_instances.ts:28`).
- `database_instance_credentials` — holds encrypted personal access tokens for PCD Git access, while
  triggers prevent writing plaintext in `database_instances`
  (`packages/praxrr-app/src/lib/server/db/migrations/20260222_encrypt_database_pat.ts:24`).
- `arr_instances` — stores each Arr instance’s URL/type/api key and metadata needed for sync targets
  (`packages/praxrr-app/src/lib/server/db/migrations/001_create_arr_instances.ts:27`).
- `arr_sync_quality_profiles`, `arr_sync_delay_profiles`, `arr_sync_media_management` (plus their
  config tables) — model which PCD databases/operations should be pushed to which Arr instance and
  under what trigger
  (`packages/praxrr-app/src/lib/server/db/migrations/015_create_arr_sync_tables.ts:20`).
- `arr_instance_credentials` — encrypted Arr API keys referenced by `arr_instances`, injected into
  clients only as needed
  (`packages/praxrr-app/src/lib/server/db/migrations/20260221_encrypt_arr_api_keys.ts:81`).
- `job_queue`/`job_run_history` — durable queue for sync/pull jobs that `triggerSyncs` and the
  startup pull flow populate
  (`packages/praxrr-app/src/lib/server/db/migrations/049_create_job_queue.ts:7`).

### Schema Details

- Operations are layered: schema scripts come from dependencies, base ops are stored in `pcd_ops`
  (published/draft states), tweaks live in the repo’s `/tweaks`, and user ops sit in `pcd_ops` as
  well (`packages/praxrr-app/src/lib/server/pcd/ops/loadOps.ts:1`). The loader preserves ordering
  when `PCDCache` builds the in-memory DB
  (`packages/praxrr-app/src/lib/server/pcd/database/cache.ts:38`), executes the SQL, and writes
  statuses into `pcd_op_history` while evaluating auto-align and guard mismatches (“value guard
  prototype” per `research/data-schema/report.md:85`).
- `OperationMetadata` surfaced by writers includes `operation`, `entity`, `name`, `stableKey`,
  `desiredState`, etc., so conflict detection can inspect what was meant versus what actually
  happened (`packages/praxrr-app/src/lib/server/pcd/core/types.ts:38`). Export/import handlers rely
  on this metadata so they can round-trip to a JSON/YAML schema (Phase 2 in
  `research/data-schema/synthesis/decision-framework.md:233`), validate with `validatePortableData`,
  and write SQL via `deserialize`.
- Base operations are re-imported from `.sql` files with metadata comments, hashing, and `sequence`
  numbers to keep `pcd_ops` in sync with upstream changes
  (`packages/praxrr-app/src/lib/server/pcd/ops/importBaseOps.ts:51`).
- The research report insists the DDL stay SQL (“SQL schema (DDL) must stay SQL” consensus,
  `research/data-schema/report.md:50`) and ranks the hybrid JSON/SQL approach highest unless the
  Phase 3 value guard prototype fails, in which case Option C/D (JSON exchange only) is still viable
  (`research/data-schema/report.md:21`, `research/data-schema/report.md:85`). Additional decision
  guidance from `research/data-schema/synthesis/decision-framework.md:233` reinforces Phase 2’s
  focus on JSON import/export plus a TRaSH adapter.
- Conflict detection/auto-align logic uses `pcdOpHistory` entries to track guard mismatches and then
  optionally drops or flags the offending operations, supporting the attack surface the report warns
  is the riskiest part (`packages/praxrr-app/src/lib/server/pcd/database/cache.ts:69`).

## External Services

- **Git-hosted PCD repositories** — `pcdManager.link` clones the repo, validates the manifest,
  processes dependencies, stores metadata, seeds built-in ops, and compiles caches;
  `pcdManager.sync` pulls updates, reimports base ops, recompiles, and fires `triggerSyncs`
  (`packages/praxrr-app/src/lib/server/pcd/core/manager.ts:39`,
  `packages/praxrr-app/src/lib/server/pcd/core/manager.ts:182`). Default auto-linking uses the
  `PRAXRR_DEFAULT_DB_*` env vars (`packages/praxrr-app/src/hooks.server.ts:52`), and the PAT is
  encrypted/stored with `database_instance_credentials`
  (`packages/praxrr-app/src/lib/server/db/migrations/20260222_encrypt_database_pat.ts:24`).
- **TRaSH-Guides** — the research strongly suggests treating TRaSH as an import adapter rather than
  forcing field-level compatibility (`research/data-schema/synthesis/contradiction-mapping.md:77`,
  `research/data-schema/synthesis/risk-assessment.md:207`). Phase 2 explicitly targets TRaSH
  import/export via JSON without changing the SQL runtime (`research/data-schema/report.md:102`).
  Persona work (see the contrarian perspective) reinforces that TRaSH’s ecosystem grip is real but
  brittle, so we should consume it without tightly coupling the format
  (`research/praxrr-additional-features/persona-findings/contrarian.md:143`).
- **Arr APIs** — sync triggers queue jobs that reach back to Radarr/Sonarr/etc. via
  `arrInstanceClients`, which decrypt API keys from `arr_instance_credentials` and cache the client
  per `(instanceId,keyVersion)`
  (`packages/praxrr-app/src/lib/server/utils/arr/arrInstanceClients.ts:54`). `triggerSyncs` enqueues
  jobs whenever PCD pulls or cache changes, so the job queue is the handoff point to Arr HTTP
  clients (`packages/praxrr-app/src/lib/server/sync/processor.ts:383`,
  `packages/praxrr-app/src/lib/server/db/migrations/049_create_job_queue.ts:7`). Arr API keys rely
  on the `ARR_CREDENTIAL_*` environment variables documented in the encryption key ring
  (`packages/praxrr-app/src/lib/server/utils/encryption/keys.ts:3`).

## Internal Services

- **PCD Manager** orchestrates linking, syncing, importing base ops, seeding built-in ops, compiling
  caches, and triggering Arr syncs, so any migration work must hook into its flow
  (`packages/praxrr-app/src/lib/server/pcd/core/manager.ts:35`,
  `packages/praxrr-app/src/lib/server/pcd/core/manager.ts:179`,
  `packages/praxrr-app/src/lib/server/pcd/core/manager.ts:347`).
- **PCD Cache** executes layered operations in memory, tracks operation results, and records
  conflicts to `pcd_op_history`, which is where value guards surface
  (`packages/praxrr-app/src/lib/server/pcd/database/cache.ts:38`).
- **Serialization/deserialization** helpers translate between the portable schema and SQL commands
  when import/export endpoints run; they cover custom formats, quality profiles, naming, media
  settings, and Lidarr metadata (`packages/praxrr-app/src/lib/server/pcd/entities/serialize.ts:1`,
  `packages/praxrr-app/src/lib/server/pcd/entities/deserialize.ts:1`).
- **Operation writer/queries** insert records into `pcd_ops` with metadata, hash, desired states,
  and sequences, so migrations that add JSON/YAML importers ultimately produce SQL operations stored
  in that table (`packages/praxrr-app/src/lib/server/db/queries/pcdOps.ts:1`). The `pcdOps` table
  and its indexes are the source of truth for what SQL runs during compilation
  (`packages/praxrr-app/src/lib/server/db/migrations/041_create_pcd_ops.ts:35`).
- **Sync processor** listens for `on_pull`/`on_change` events, sets sync configs to pending, and
  schedules Arr jobs via the job queue (`packages/praxrr-app/src/lib/server/sync/processor.ts:383`).
  The job queue service enqueues/deduplicates jobs for quality profiles, delay profiles, media
  management, and metadata profiles (`packages/praxrr-app/src/lib/server/jobs/queueService.ts:1`).
- **Arr client factory** decrypts Arr API keys on demand so they are never stored or logged in
  plaintext, and caching avoids repeated decryption for the same key version
  (`packages/praxrr-app/src/lib/server/utils/arr/arrInstanceClients.ts:54`). These clients feed the
  sync sections that run after migrations or exports.

## Configuration

- **PCD defaults & job control** — `APP_BASE_PATH`/`TZ`/`PARSER_HOST`/`PARSER_PORT`/`PORT`/`HOST`
  plus feature flags like `PRAXRR_VALIDATE_INSTANCES`, `PULL_ON_START`,
  `PULL_ON_START_MAX_CONCURRENCY`, and `PULL_ON_START_TIMEOUT_MS` come from
  `packages/praxrr-app/src/lib/server/utils/config/config.ts:31`. `hooks.server.ts:52` uses the
  `PRAXRR_DEFAULT_DB_*` variables (repo URL/branch/name/token/Git identity) to auto-link a default
  PCD on first run, and `config.pullOnStart` controls whether the startup pull job fires
  (`packages/praxrr-app/src/hooks.server.ts:131`).
- **Authentication** — the `AUTH` mode influences `hooks.server.ts:185`, where `handle` uses
  `getAuthState` (`packages/praxrr-app/src/lib/server/utils/auth/middleware.ts:37`). Public API
  paths (e.g., `/api/v1/health`) are whitelisted, while other calls require an API key or session,
  and unauthorized API tries receive a 401.
- **Secrets encryption** — Git PATs and Arr API keys rely on AES-GCM keys loaded from
  `ARR_CREDENTIAL_MASTER_KEY`, `ARR_CREDENTIAL_MASTER_KEY_VERSION`, and
  `ARR_CREDENTIAL_PREVIOUS_KEYS` (`packages/praxrr-app/src/lib/server/utils/config/config.ts:72`,
  `packages/praxrr-app/src/lib/server/utils/encryption/keys.ts:3`). The encryption helpers reject
  malformed base64 and enforce key versions before decrypting
  (`packages/praxrr-app/src/lib/server/utils/encryption/keys.ts:3`), so any migration that
  introduces new secrets must also respect that key ring.

This research combines the existing API surface, the SQL-first schema tracked in the
`pcd_ops`/`pcd_op_history` tables, and the enforcement of encrypted credentials/jobs so that
`pcd-data-migration` can safely expand the import/export experience while preserving the value
guards that the February 22, 2026 report flags as the hardest problem
(`research/data-schema/report.md:21`, `research/data-schema/report.md:50`).
