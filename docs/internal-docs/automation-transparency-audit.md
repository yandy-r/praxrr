# Automation Transparency Audit

- **Audit date:** 2026-07-09
- **Scope:** every production `JobType` plus material direct automation and evidence workflows required by issue #21
- **Registry source:** `packages/praxrr-app/src/lib/server/jobs/queueTypes.ts` and `packages/praxrr-app/src/lib/server/jobs/transparencyAudit.ts`

## Reading the audit

This is an evidence audit, not a claim that every operation has entity-level confirmation or exact field provenance.

- **Planned** means a preview or diff computed before a write. It never proves the write occurred.
- **Actual** means evidence recorded after an operation. Its stated granularity matters: a job, section, instance, operation count, or entity outcome is not interchangeable.
- **Pass** means the current evidence is sufficient at the stated granularity.
- **Not Applicable** means the discovered path is not an automation workflow, with the reason recorded.
- **Follow-up** identifies an unresolved gap by a stable label. Each follow-up is linked to its templated engineering issue.
- Logger metadata is sanitized by `packages/praxrr-app/src/lib/server/utils/logger/logger.ts`. Durable `job_run_history.error` and `output` strings are not passed through that sanitizer before persistence or display; the queued rows therefore share [GAP-QUEUED-JOB-DURABLE-EVIDENCE-SAFETY](https://github.com/yandy-r/praxrr/issues/237).

The compile-time and runtime coverage guard is `packages/praxrr-app/src/tests/jobs/transparencyAudit.test.ts`: it compares `JOB_TYPES`, audit keys, and registered handlers. It does not validate the truth of prose or sanitize durable output, so this document cites the relevant implementation and tests.

Path shorthand in the matrices is repo-relative and unambiguous: `jobs/`, `sync/`, `pcd/`, `health/`, and `goals/` start at `packages/praxrr-app/src/lib/server/`; `api/v1/` and other page paths start at `packages/praxrr-app/src/routes/`; and `tests/` starts at `packages/praxrr-app/src/`.

## Planned-versus-actual limits

1. Sync Preview create, Quality Goals preview, rollback preview, resolved live diff, drift comparison, and canary remaining-target previews are read-only computations. Their entity rows are planned or observed diffs, not confirmed writes.
2. Sync Preview apply calls `executeSyncJob` again. The immediate result and Sync History prove job/section outcomes and item counts, but current syncer contracts do not return one terminal result per attempted entity. [GAP-SYNC-CONFIRMED-ENTITY-OUTCOMES](https://github.com/yandy-r/praxrr/issues/232) owns that prerequisite.
3. Resolved Config proves base-side state, the base-versus-resolved user diff, final resolved state, and pending-conflict ambiguity. It does not retain schema-default/base-op/tweaks-op/user-op last-writer lineage. [GAP-RESOLVED-EXACT-FIELD-LINEAGE](https://github.com/yandy-r/praxrr/issues/231) owns that prerequisite.
4. Rollback success is actual at op-rewind and post-verification granularity. Its pre-rollback entity diff remains a plan; it is not restyled as a confirmed per-entity result.
5. Canary detail records actual canary and per-target job results. Its remaining-target diff is representative planned evidence only.

## Queued workflow matrix

All queued jobs inherit the dispatcher contract in `packages/praxrr-app/src/lib/server/jobs/dispatcher.ts`: queue source, run time, status, duration, error/output, and reschedule time are recorded. `/settings/jobs` renders status/timing and a one-line error/output cell through `JobHistory.svelte`. Because handler exceptions and handler-controlled strings are stored verbatim, every queued row is a linked follow-up for the shared durable-evidence safety gap even when its domain evidence is otherwise complete.
