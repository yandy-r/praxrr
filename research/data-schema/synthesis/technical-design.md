# Technical Design: Hybrid JSON/YAML + SQL PCD System

## Overview

This document specifies the concrete implementation of the hybrid model identified by research
convergence: JSON/YAML for entity authoring and seed data, SQL for schema DDL and incremental
mutation ops, with a compiler that bridges the two. The runtime PCD cache (in-memory SQLite) is
unchanged.

### Design Principles

1. **Portable types are the schema.** The existing `portable.ts` types define the JSON/YAML
   structure. Do not invent a new entity model.
2. **SQL ops remain the execution format.** JSON/YAML is compiled to SQL ops before entering the
   existing pipeline. The append-only model, value guards, and conflict detection are preserved.
3. **Seed data and incremental mutations are different problems.** JSON/YAML replaces only the seed
   layer. Incremental mutations stay as SQL ops (or are generated by the compiler from entity diffs).
4. **The compiler compiles down, never up.** JSON/YAML -> SQL is the only direction. SQL -> JSON/YAML
   is a separate serialization concern (already handled by `serialize.ts`).

## 1. Entity JSON/YAML Format

### 1.1 File Organization

The `praxrr-db` repository would gain an `entities/` directory alongside the existing `ops/`
directory. The structure uses **one file per entity**, organized by entity type:

```
praxrr-db/
  pcd.json                          # manifest (unchanged)
  ops/                              # existing SQL ops (incremental mutations)
    0.rosettarr.sql                 # DEPRECATED: replaced by entities/ for seed
    1.update-include-amzn-cf.sql    # incremental mutation (stays as SQL)
    ...
  entities/                         # NEW: JSON/YAML entity definitions (seed data)
    tags.yaml                       # all tags in one file (simple list entity)
    regular-expressions/
      AVC.yaml
      HEVC.yaml
      Amazon-Prime.yaml
      DTS-HD-MA.yaml
      ...
    custom-formats/
      AMZN.yaml
      DTS-HD-MA.yaml
      1080p-Balanced-Tier-1.yaml
      Extras.yaml
      Banned-Groups.yaml
      ...
    quality-profiles/
      1080p-Balanced.yaml
      1080p-Quality.yaml
      2160p-Remux.yaml
      ...
    delay-profiles/
      Default.yaml
    quality-api-mappings.yaml       # all API mappings in one file
    media-management/
      radarr-naming/
        Default.yaml
      sonarr-naming/
        Default.yaml
      radarr-media-settings/
        Default.yaml
      sonarr-media-settings/
        Default.yaml
      radarr-quality-definitions/
        Default.yaml
      sonarr-quality-definitions/
        Default.yaml
      lidarr-naming/
        Default.yaml
      lidarr-media-settings/
        Default.yaml
      lidarr-quality-definitions/
        Default.yaml
    metadata-profiles/
      lidarr/
        Default.yaml
```

**Rationale for one-file-per-entity:**

- Git diffs show exactly which entity changed.
- PRs adding a new custom format touch one file.
- Filenames are slugified entity names (kebab-case, special characters stripped).
- Small entities (tags, quality API mappings) use a single list file because they have no internal
  structure worth splitting.

### 1.2 Regular Expression Entity

A regular expression is the simplest compound entity: a self-contained definition with optional tags.

```yaml
# entities/regular-expressions/DTS-HD-MA.yaml
name: 'DTS-HD MA'
pattern: '\b(dts[-_. ]?(ma|hd([-_. ]?ma)?|xll))(\b|\d)'
description: ''
regex101Id: null
tags:
  - Audio
  - Lossless Audio
```

**Field mapping to schema:**

| YAML field    | Table                     | Column                    |
| ------------- | ------------------------- | ------------------------- |
| `name`        | `regular_expressions`     | `name`                    |
| `pattern`     | `regular_expressions`     | `pattern`                 |
| `description` | `regular_expressions`     | `description`             |
| `regex101Id`  | `regular_expressions`     | `regex101_id`             |
| `tags[]`      | `regular_expression_tags` | `tag_name` (FK to `tags`) |

### 1.3 Custom Format Entity

Custom formats are the most complex entity, spanning up to 11 tables. The YAML format mirrors the
existing `PortableCustomFormat` type with `ConditionData` inlined.

```yaml

description: "Matches 'Amazon Prime' WEB-DLs. Negates any encodes."

  - Streaming Service

  - name: 'Amazon Prime'

    arrType: all
```

### 1.9 Cross-Entity References

All cross-entity references use **name strings**, matching the existing PCD schema convention:

- A `condition_patterns` entry references a `regular_expressions` entity by
  `patterns[].name` -> `regular_expression_name`.
- A `quality_profile_custom_formats` entry references a `custom_formats` entity by
  `customFormatScores[].customFormatName` -> `custom_format_name`.
- A `quality_group_members` entry references a `qualities` entity by
  `orderedItems[].members[].name` -> `quality_name`.

**There are no numeric IDs in the YAML format.** This matches the `portable.ts` design and the PCD
schema's name-based foreign keys.

## 2. The Compiler

### 2.1 Architecture

```
entities/                          ops/
  *.yaml files  -----+             *.sql files (incremental)
                     |                  |
                     v                  |
            +------------------+        |
            |  YAML Loader     |        |
            |  - parse YAML    |        |
            |  - validate      |        |
            +--------+---------+        |
                     |                  |
                     v                  |
            +------------------+        |
            |  Entity Graph    |        |
            |  - resolve deps  |        |
            |  - topological   |        |
            |    sort           |        |
            +--------+---------+        |
                     |                  |
                     v                  |
            +------------------+        |
            |  SQL Generator   |        |
            |  - entity -> SQL |        |
            |  - INSERT stmts  |        |
            +--------+---------+        |
                     |                  |
                     v                  v
            +-----------------------------------+
            |  pcd_ops (append-only SQL store)   |
            +-----------------------------------+
                     |
                     v
            +-----------------------------------+
            |  In-Memory SQLite Cache (build)    |
            +-----------------------------------+
```

### 2.2 Compilation Pipeline

The compiler transforms `entities/*.yaml` into a single SQL script that replaces
`0.rosettarr.sql`. This happens at **PCD build time** (when importing a database), not at
authoring time.

```typescript
interface CompilerInput {
  entityDir: string; // path to entities/ directory
  schemaPath: string; // path to 0.schema.sql
}

interface CompilerOutput {
  sql: string; // complete SQL script (equivalent to 0.rosettarr.sql)
  entityCount: number; // number of entities compiled
  warnings: string[]; // non-fatal issues
}

async function compileEntitiesToSql(input: CompilerInput): Promise<CompilerOutput> {
  // 1. Load and parse all YAML files
  const entities = await loadAllEntities(input.entityDir);

  // 2. Validate entity references
  const validationErrors = validateEntityGraph(entities);
  if (validationErrors.length > 0) {
    throw new CompilationError(validationErrors);
  }

  // 3. Topological sort by dependency order
  const sorted = topologicalSort(entities);

  // 4. Generate SQL for each entity in dependency order
  const sqlStatements: string[] = [];
  for (const entity of sorted) {
    sqlStatements.push(...generateSqlForEntity(entity));
  }

  return {
    sql: sqlStatements.join('\n'),
    entityCount: sorted.length,
    warnings: [],
  };
}
```

### 2.3 Dependency Resolution

Entities must be inserted in FK order. The dependency graph is static and known from the schema:

```
Level 0 (no dependencies):
  tags, languages, qualities

Level 1 (depends on Level 0):
  regular_expressions          (tags via regular_expression_tags)
  quality_api_mappings         (qualities)

Level 2 (depends on Level 1):
  custom_formats               (tags via custom_format_tags)

Level 3 (depends on Level 2):
  custom_format_conditions     (custom_formats)
  condition_patterns           (custom_format_conditions, regular_expressions)
  condition_languages          (custom_format_conditions, languages)
  condition_sources            (custom_format_conditions)
  condition_resolutions        (custom_format_conditions)
  condition_quality_modifiers  (custom_format_conditions)
  condition_release_types      (custom_format_conditions)
  condition_indexer_flags       (custom_format_conditions)
  condition_sizes              (custom_format_conditions)
  condition_years              (custom_format_conditions)
  custom_format_tests          (custom_formats)

Level 4 (depends on Level 2):
  quality_profiles             (tags via quality_profile_tags)

Level 5 (depends on Level 4):
  quality_groups               (quality_profiles)
  quality_profile_languages    (quality_profiles, languages)
  quality_profile_custom_formats (quality_profiles, custom_formats)

Level 6 (depends on Level 5):
  quality_group_members        (quality_groups, qualities)
  quality_profile_qualities    (quality_profiles, qualities, quality_groups)

Level 7 (standalone):
  delay_profiles
  *_naming
  *_media_settings
  *_quality_definitions
  lidarr_metadata_profiles     (and child tables)
```

The compiler emits SQL in this order. Within each level, entities are sorted alphabetically by name
for deterministic output.

### 2.4 SQL Generation Per Entity Type

**Regular Expression:**

```typescript
function generateRegexSql(re: PortableRegularExpression): string[] {
  const sqls: string[] = [];

  // Ensure tags exist (idempotent)
  for (const tag of re.tags) {
    sqls.push(`INSERT OR IGNORE INTO tags (name) VALUES ('${esc(tag)}');`);
  }

  // Insert the regex
  sqls.push(
    `INSERT INTO regular_expressions (name, pattern, description, regex101_id) ` +
      `VALUES ('${esc(re.name)}', '${esc(re.pattern)}', '${esc(re.description ?? '')}', ` +
      `${re.regex101Id ? `'${esc(re.regex101Id)}'` : 'NULL'});`
  );

  // Insert tag junctions
  for (const tag of re.tags) {
    sqls.push(
      `INSERT INTO regular_expression_tags (regular_expression_name, tag_name) ` +
        `VALUES ('${esc(re.name)}', '${esc(tag)}');`
    );
  }

  return sqls;
}
```

**Custom Format (with polymorphic conditions):**

```typescript
function generateCustomFormatSql(cf: PortableCustomFormat): string[] {
  const sqls: string[] = [];

  // Ensure tags exist
  for (const tag of cf.tags) {
    sqls.push(`INSERT OR IGNORE INTO tags (name) VALUES ('${esc(tag)}');`);
  }

  // Insert the custom format
  sqls.push(
    `INSERT INTO custom_formats (name, description, include_in_rename) ` +
      `VALUES ('${esc(cf.name)}', '${esc(cf.description ?? '')}', ${cf.includeInRename ? 1 : 0});`
  );

  // Insert tag junctions
  for (const tag of cf.tags) {
    sqls.push(
      `INSERT INTO custom_format_tags (custom_format_name, tag_name) ` + `VALUES ('${esc(cf.name)}', '${esc(tag)}');`
    );
  }

  // Insert conditions (polymorphic dispatch)
  for (const cond of cf.conditions) {
    sqls.push(
      `INSERT INTO custom_format_conditions ` +
        `(custom_format_name, name, type, arr_type, negate, required) ` +
        `VALUES ('${esc(cf.name)}', '${esc(cond.name)}', '${esc(cond.type)}', ` +
        `'${esc(cond.arrType || 'all')}', ${cond.negate ? 1 : 0}, ${cond.required ? 1 : 0});`
    );

    // Type-specific value table
    switch (cond.type) {
      case 'release_title':
      case 'release_group':
      case 'edition':
        for (const pattern of cond.patterns ?? []) {
          sqls.push(
            `INSERT INTO condition_patterns ` +
              `(custom_format_name, condition_name, regular_expression_name) ` +
              `VALUES ('${esc(cf.name)}', '${esc(cond.name)}', '${esc(pattern.name)}');`
          );
        }
        break;

      case 'language':
        for (const lang of cond.languages ?? []) {
          sqls.push(
            `INSERT INTO condition_languages ` +
              `(custom_format_name, condition_name, language_name, except_language) ` +
              `VALUES ('${esc(cf.name)}', '${esc(cond.name)}', '${esc(lang.name)}', ` +
              `${lang.except ? 1 : 0});`
          );
        }
        break;

      case 'source':
        for (const source of cond.sources ?? []) {
          sqls.push(
            `INSERT INTO condition_sources ` +
              `(custom_format_name, condition_name, source) ` +
              `VALUES ('${esc(cf.name)}', '${esc(cond.name)}', '${esc(source)}');`
          );
        }
        break;

      case 'resolution':
        for (const resolution of cond.resolutions ?? []) {
          sqls.push(
            `INSERT INTO condition_resolutions ` +
              `(custom_format_name, condition_name, resolution) ` +
              `VALUES ('${esc(cf.name)}', '${esc(cond.name)}', '${esc(resolution)}');`
          );
        }
        break;

      case 'quality_modifier':
        for (const qm of cond.qualityModifiers ?? []) {
          sqls.push(
            `INSERT INTO condition_quality_modifiers ` +
              `(custom_format_name, condition_name, quality_modifier) ` +
              `VALUES ('${esc(cf.name)}', '${esc(cond.name)}', '${esc(qm)}');`
          );
        }
        break;

      case 'release_type':
        for (const rt of cond.releaseTypes ?? []) {
          sqls.push(
            `INSERT INTO condition_release_types ` +
              `(custom_format_name, condition_name, release_type) ` +
              `VALUES ('${esc(cf.name)}', '${esc(cond.name)}', '${esc(rt)}');`
          );
        }
        break;

      case 'indexer_flag':
        for (const flag of cond.indexerFlags ?? []) {
          sqls.push(
            `INSERT INTO condition_indexer_flags ` +
              `(custom_format_name, condition_name, flag) ` +
              `VALUES ('${esc(cf.name)}', '${esc(cond.name)}', '${esc(flag)}');`
          );
        }
        break;

      case 'size':
        if (cond.size) {
          sqls.push(
            `INSERT INTO condition_sizes ` +
              `(custom_format_name, condition_name, min_bytes, max_bytes) ` +
              `VALUES ('${esc(cf.name)}', '${esc(cond.name)}', ` +
              `${cond.size.minBytes ?? 'NULL'}, ${cond.size.maxBytes ?? 'NULL'});`
          );
        }
        break;

      case 'year':
        if (cond.years) {
          sqls.push(
            `INSERT INTO condition_years ` +
              `(custom_format_name, condition_name, min_year, max_year) ` +
              `VALUES ('${esc(cf.name)}', '${esc(cond.name)}', ` +
              `${cond.years.minYear ?? 'NULL'}, ${cond.years.maxYear ?? 'NULL'});`
          );
        }
        break;
    }
  }

  // Insert tests
  for (const test of cf.tests) {
    sqls.push(
      `INSERT INTO custom_format_tests ` +
        `(custom_format_name, title, type, should_match, description) ` +
        `VALUES ('${esc(cf.name)}', '${esc(test.title)}', '${esc(test.type)}', ` +
        `${test.shouldMatch ? 1 : 0}, ${test.description ? `'${esc(test.description)}'` : 'NULL'});`
    );
  }

  return sqls;
}
```

### 2.5 Validation Before Compilation

The compiler validates the entity graph before generating SQL:

```typescript
interface ValidationError {
  entityType: string;
  entityName: string;
  field: string;
  message: string;
}

function validateEntityGraph(entities: AllEntities): ValidationError[] {
  const errors: ValidationError[] = [];
  const tagNames = new Set(entities.tags);
  const regexNames = new Set(entities.regularExpressions.map((r) => r.name));
  const cfNames = new Set(entities.customFormats.map((c) => c.name));
  const qualityNames = new Set(entities.qualities); // from schema seed
  const languageNames = new Set(entities.languages); // from schema seed

  // Validate regex tag references
  for (const re of entities.regularExpressions) {
    for (const tag of re.tags) {
      if (!tagNames.has(tag)) {
        errors.push({
          entityType: 'regular_expression',
          entityName: re.name,
          field: 'tags',
          message: `Tag "${tag}" does not exist`,
        });
      }
    }
  }

  // Validate CF condition references
  for (const cf of entities.customFormats) {
    for (const cond of cf.conditions) {
      // Validate pattern references point to existing regexes
      if (cond.patterns) {
        for (const pattern of cond.patterns) {
          if (!regexNames.has(pattern.name)) {
            errors.push({
              entityType: 'custom_format',
              entityName: cf.name,
              field: `conditions.${cond.name}.patterns`,
              message: `Regular expression "${pattern.name}" does not exist`,
            });
          }
        }
      }

      // Validate language references
      if (cond.languages) {
        for (const lang of cond.languages) {
          if (!languageNames.has(lang.name)) {
            errors.push({
              entityType: 'custom_format',
              entityName: cf.name,
              field: `conditions.${cond.name}.languages`,
              message: `Language "${lang.name}" does not exist`,
            });
          }
        }
      }
    }
  }

  // Validate QP score references
  for (const qp of entities.qualityProfiles) {
    for (const score of qp.customFormatScores) {
      if (!cfNames.has(score.customFormatName)) {
        errors.push({
          entityType: 'quality_profile',
          entityName: qp.name,
          field: `customFormatScores`,
          message: `Custom format "${score.customFormatName}" does not exist`,
        });
      }
    }

    // Validate quality references in orderedItems
    for (const item of qp.orderedItems) {
      if (item.type === 'quality' && !qualityNames.has(item.name)) {
        errors.push({
          entityType: 'quality_profile',
          entityName: qp.name,
          field: `orderedItems`,
          message: `Quality "${item.name}" does not exist`,
        });
      }
      if (item.members) {
        for (const member of item.members) {
          if (!qualityNames.has(member.name)) {
            errors.push({
              entityType: 'quality_profile',
              entityName: qp.name,
              field: `orderedItems.${item.name}.members`,
              message: `Quality "${member.name}" does not exist`,
            });
          }
        }
      }
    }
  }

  // Case-insensitive uniqueness check
  const namesByType = new Map<string, string[]>();
  // ... check for duplicate names within each entity type

  return errors;
}
```

### 2.6 Value-Guarded UPDATE Ops (Incremental Changes)

**Incremental mutations stay as SQL ops.** This is the core insight from the convergence analysis:
the seed data/incremental migration distinction is the natural split point.

However, the compiler can assist with generating incremental ops from entity diffs. When a
contributor modifies a YAML entity file and pushes to the database repository, a CI pipeline can:

1. Compile the previous version of the entity to SQL and execute against the schema
2. Compile the new version of the entity to SQL
3. Diff the two compiled states to generate incremental SQL ops with value guards

```typescript
interface IncrementalOp {
  sql: string;
  metadata: {
    operation: 'update' | 'delete';
    entity: string;
    name: string;
    changedFields?: string[];
  };
}

async function generateIncrementalOps(
  oldEntity: PortableCustomFormat,
  newEntity: PortableCustomFormat
): Promise<IncrementalOp[]> {
  const ops: IncrementalOp[] = [];

  // Detect field changes on the parent entity
  if (oldEntity.includeInRename !== newEntity.includeInRename) {
    ops.push({
      sql:
        `UPDATE "custom_formats" SET "include_in_rename" = ${newEntity.includeInRename ? 1 : 0} ` +
        `WHERE "name" = '${esc(newEntity.name)}' ` +
        `AND "include_in_rename" = ${oldEntity.includeInRename ? 1 : 0};`,
      metadata: {
        operation: 'update',
        entity: 'custom_format',
        name: newEntity.name,
        changedFields: ['include_in_rename'],
      },
    });
  }

  if (oldEntity.description !== newEntity.description) {
    ops.push({
      sql:
        `UPDATE "custom_formats" SET "description" = '${esc(newEntity.description ?? '')}' ` +
        `WHERE "name" = '${esc(newEntity.name)}' ` +
        `AND "description" = '${esc(oldEntity.description ?? '')}';`,
      metadata: {
        operation: 'update',
        entity: 'custom_format',
        name: newEntity.name,
        changedFields: ['description'],
      },
    });
  }

  // Detect added/removed conditions
  // ... (diff conditions arrays, generate INSERT/DELETE with appropriate guards)

  // Detect added/removed tags
  // ... (diff tags arrays, generate INSERT/DELETE for junction table)

  return ops;
}
```

**Key insight:** The value guard in the generated SQL (`AND "include_in_rename" = 0`) comes from the
`oldEntity` state. This preserves the conflict detection semantics: if a user has overridden the
field, the guard fails, 0 rows are affected, and the existing conflict detection in `cache.ts`
handles it identically to hand-written SQL ops.

### 2.7 Git Diff to SQL Ops Conversion

For the CI pipeline that generates incremental ops from entity file changes:

```
Git push with modified entities/custom-formats/AMZN.yaml
    |
    v
CI pipeline:
    1. git diff HEAD~1..HEAD -- entities/
    2. For each changed file:
       a. Load old version: git show HEAD~1:entities/custom-formats/AMZN.yaml
       b. Load new version: entities/custom-formats/AMZN.yaml
       c. Parse both to PortableCustomFormat
       d. Call generateIncrementalOps(old, new)
       e. Write SQL ops to ops/{N}.{slug}.sql
    3. git add ops/ && git commit
```

This flow is **optional and additive**. Contributors can still write SQL ops by hand for complex
migrations. The CI pipeline automates the common case (simple field changes, added/removed
conditions, score adjustments).

## 3. The Migration Path

### 3.1 Converting 25,000 Lines of Existing SQL to YAML

The migration tool uses the existing `serialize.ts` infrastructure. After compiling all 57 existing
SQL ops files into the in-memory cache, it serializes every entity to its portable form, then
writes YAML files:

```typescript
async function migrateToYaml(cache: PCDCache, outputDir: string): Promise<void> {
  // 1. Serialize all tags
  const tags = await listAllTags(cache);
  await writeYaml(`${outputDir}/tags.yaml`, { tags: tags.map((t) => t.name) });

  // 2. Serialize all regular expressions
  const regexes = await listAllRegexNames(cache);
  for (const name of regexes) {
    const portable = await serializeRegularExpression(cache, name);
    const slug = slugify(name);
    await writeYaml(`${outputDir}/regular-expressions/${slug}.yaml`, portable);
  }

  // 3. Serialize all custom formats
  const formats = await listAllCfNames(cache);
  for (const name of formats) {
    const portable = await serializeCustomFormat(cache, name);
    const slug = slugify(name);
    await writeYaml(`${outputDir}/custom-formats/${slug}.yaml`, portable);
  }

  // 4. Serialize all quality profiles
  const profiles = await listAllQpNames(cache);
  for (const name of profiles) {
    const portable = await serializeQualityProfile(cache, name);
    const slug = slugify(name);
    await writeYaml(`${outputDir}/quality-profiles/${slug}.yaml`, portable);
  }

  // 5. Serialize all other entity types...
}
```

**This is a one-time conversion.** The tool:

1. Loads the PCD schema (`0.schema.sql`)
2. Loads the language/quality seeds (`1.languages.sql`, `2.qualities.sql`)
3. Loads all 57 base ops files in order
4. Compiles the full cache
5. Serializes every entity to YAML using the existing `serialize.ts` functions
6. Writes the YAML files to the `entities/` directory

### 3.2 Handling the 55 Incremental Migration Files

The 55 incremental ops files (`1.*.sql` through `56.*.sql`) represent the **history** of changes.
After migration:

- `entities/*.yaml` represents the **current state** (equivalent to replaying all 57 ops)
- `ops/0.rosettarr.sql` is replaced by the compiled output of `entities/`
- `ops/1.*.sql` through `ops/56.*.sql` **remain in the repository** as historical record
- New incremental changes are added as new `ops/{N}.*.sql` files, generated from entity diffs

The ops directory becomes append-only history, while the entities directory is the current
canonical state. The compilation order is:

1. Schema (from `deps/schema/ops/`)
2. Seed (compiled from `entities/`, equivalent to what was `0.rosettarr.sql`)
3. Incremental ops (from `ops/57.*.sql` onward -- ops 0-56 are embedded in the seed)
4. Tweaks (from `tweaks/`)
5. User ops (from `pcd_ops` table)

### 3.3 Proving Identical Runtime State

The migration must produce bit-for-bit identical compiled cache state. The verification strategy:

```typescript
async function verifyMigration(
  schemaPath: string,
  oldOpsDir: string, // contains 0.rosettarr.sql + all incremental ops
  newEntitiesDir: string // contains entities/*.yaml
): Promise<boolean> {
  // 1. Build cache from old format (SQL ops replay)
  const oldCache = buildCacheFromSqlOps(schemaPath, oldOpsDir);

  // 2. Build cache from new format (YAML compile + remaining SQL ops)
  const compiledSeed = await compileEntitiesToSql({ entityDir: newEntitiesDir, schemaPath });
  const newCache = buildCacheFromCompiledSeed(schemaPath, compiledSeed);

  // 3. Compare every table row-by-row
  const tables = listAllTables(oldCache);
  for (const table of tables) {
    const oldRows = queryAllRows(oldCache, table);
    const newRows = queryAllRows(newCache, table);

    // Sort by primary key for deterministic comparison
    // Compare row counts, then field values
    // Ignore auto-increment IDs and timestamps
    if (!rowsMatch(oldRows, newRows, { ignoreColumns: ['id', 'created_at', 'updated_at'] })) {
      console.error(`Table ${table}: mismatch`);
      return false;
    }
  }

  return true;
}
```

**What is compared:**

- All 33+ tables
- Row counts per table
- Field values (excluding auto-increment IDs and timestamps)
- Foreign key integrity (the new cache must pass all FK checks)

**What is not compared:**

- `id` columns (auto-increment values depend on insertion order)
- `created_at` / `updated_at` timestamps (differ per compile run)
- Row ordering within tables (compared as sets, not sequences)

### 4.3 Handling Denormalized TRaSH Data

The critical structural difference: TRaSH inlines regex patterns as string values inside each
specification. Praxrr separates regex patterns as shared entities. The adapter must deduplicate:

```typescript
interface TrashImportContext {
  // Track regex patterns seen across all TRaSH CFs to deduplicate
  regexMap: Map<string, string>; // pattern -> generated Praxrr regex name
  // Track trash_id -> Praxrr name mappings for provenance
  trashIdMap: Map<string, string>;
}

function importTrashCustomFormat(
  trashCf: TrashCustomFormat,
  ctx: TrashImportContext
): { regex: PortableRegularExpression | null; cf: PortableCustomFormat } {
  const conditions: ConditionData[] = [];
  let newRegex: PortableRegularExpression | null = null;

  for (const spec of trashCf.specifications) {
    if (spec.implementation === 'ReleaseTitleSpecification') {
      const pattern = spec.fields.value;

      // Check if we already have a regex with this pattern
      let regexName = ctx.regexMap.get(pattern);
      if (!regexName) {
        // Create a new regex entity named after the condition
        regexName = `${trashCf.name} - ${spec.name}`;
        ctx.regexMap.set(pattern, regexName);
        newRegex = {
          name: regexName,
          pattern: pattern,
          tags: [],
          description: `Imported from TRaSH-Guides (${trashCf.trash_id})`,
          regex101Id: trashCf.trash_regex ? extractRegex101Id(trashCf.trash_regex) : null,
        };
      }

      conditions.push({
        name: spec.name,
        type: 'release_title',
        arrType: '', // TRaSH CFs are arr-agnostic; scoped at import
        negate: spec.negate,
        required: spec.required,
        patterns: [{ name: regexName, pattern }],
      });
    } else {
      // Map other specification types...
      conditions.push(mapTrashSpecToCondition(spec));
    }
  }

  return {
    regex: newRegex,
    cf: {
      name: trashCf.name,
      description: null,
      includeInRename: trashCf.includeCustomFormatWhenRenaming,
      tags: [], // TRaSH does not have tags; user assigns post-import
      conditions,
      tests: [], // TRaSH does not include test cases
    },
  };
}
```

By diffing entity YAML files between git tags, the system can generate human-readable changelogs:

```typescript
interface ChangelogEntry {
  entityType: string;
  entityName: string;
  changeType: 'added' | 'removed' | 'modified';
  details: string[];
}

async function generateChangelog(oldEntities: AllEntities, newEntities: AllEntities): Promise<ChangelogEntry[]> {
  const entries: ChangelogEntry[] = [];

  // Detect added/removed custom formats
  const oldCfNames = new Set(oldEntities.customFormats.map((c) => c.name));
  const newCfNames = new Set(newEntities.customFormats.map((c) => c.name));

  for (const name of newCfNames) {
    if (!oldCfNames.has(name)) {
      const cf = newEntities.customFormats.find((c) => c.name === name)!;
      entries.push({
        entityType: 'custom_format',
        entityName: name,
        changeType: 'added',
        details: [`Tags: ${cf.tags.join(', ') || 'none'}`],
      });
    }
  }

  for (const name of oldCfNames) {
    if (!newCfNames.has(name)) {
      entries.push({
        entityType: 'custom_format',
        entityName: name,
        changeType: 'removed',
        details: [],
      });
    }
  }

  // Detect modified entities (deep diff)
  for (const name of newCfNames) {
    if (oldCfNames.has(name)) {
      const oldCf = oldEntities.customFormats.find((c) => c.name === name)!;
      const newCf = newEntities.customFormats.find((c) => c.name === name)!;
      const diffs = diffCustomFormat(oldCf, newCf);
      if (diffs.length > 0) {
        entries.push({
          entityType: 'custom_format',
          entityName: name,
          changeType: 'modified',
          details: diffs,
        });
      }
    }
  }

  // Repeat for quality profiles, regexes, etc.
  return entries;
}
```

## 7. YAML Safety Rules

Based on the research convergence on YAML risks (Convergence 10), the following strict rules apply:

1. **YAML 1.2 only.** The parser must use YAML 1.2 spec (not 1.1), which eliminates the Norway
   problem (`NO` being parsed as `false`).

2. **All string values must be quoted** when they could be ambiguous. The YAML writer enforces
   quoting for:
   - Entity names that look like booleans (`"NO"`, `"Yes"`, `"True"`, `"False"`)
   - Entity names that look like numbers (`"1080p"` -- though the `p` suffix saves this)
   - Regex patterns (always double-quoted with proper escaping)

3. **JSON Schema validation** is run on every YAML file before compilation. The JSON Schema
   is generated from the `portable.ts` types.

4. **Regex patterns use single-quoted YAML strings** where possible, since single-quoted YAML
   strings have no escape sequences (the only special character is `''` for literal single quote).
   For patterns containing single quotes, double-quoted strings with explicit escaping are used.

5. **No implicit type coercion.** The parser is configured to treat all unquoted values as strings
   unless they match the JSON Schema type. Numeric fields (`score`, `position`, etc.) are validated
   by schema.

## 9. Phased Implementation Roadmap

### Phase 1: JSON Schema Formalization (Low Risk, Immediate Value)

**Goal:** Publish a JSON Schema derived from `portable.ts` types.

**Deliverables:**

- `packages/praxrr-schema/pcd-entities.schema.json`
- Schema generation script (TypeScript types -> JSON Schema)
- IDE integration (VS Code YAML extension reads the schema)

**Impact:** Zero runtime changes. Contributors get autocomplete and validation in their editors
when editing entity files. The schema doubles as API documentation.

**Effort estimate:** 1-2 days.

### Phase 2: Migration Tool (Medium Risk)

**Goal:** Convert the existing 25K-line `0.rosettarr.sql` (plus all 56 incremental ops) into
`entities/*.yaml` files.

**Deliverables:**

- `scripts/migrate-to-yaml.ts` -- one-time migration tool
- `entities/` directory with all entity YAML files
- Verification script proving identical compiled state

**Effort estimate:** 3-5 days (most time spent on verification).

### Phase 3: Compiler (Medium Risk)

**Goal:** Build the YAML -> SQL compiler that generates seed SQL from entity files.

**Deliverables:**

- `packages/praxrr-app/src/lib/server/pcd/compiler/` -- compiler module
- Modifications to `importBaseOps.ts` to detect and compile `entities/` directory
- Unit tests covering every entity type and condition variant

**Effort estimate:** 5-8 days.

### Phase 4: CI Pipeline for Incremental Ops (Medium Risk)

**Goal:** Automate generation of SQL ops from entity YAML diffs.

**Deliverables:**

- GitHub Action that detects entity file changes and generates SQL ops
- Diff-to-ops generation for common cases (field changes, added/removed conditions)
- Human review gate before merging generated ops

**Effort estimate:** 3-5 days.

### Phase 5: TRaSH Import Adapter (Medium Risk, Independent)

**Goal:** One-way import of TRaSH-Guides custom formats into Praxrr.

**Deliverables:**

- `packages/praxrr-app/src/lib/server/pcd/adapters/trash/` -- adapter module
- API endpoint for triggering import
- UI for selecting TRaSH CFs to import
- trash_id tracking table for provenance

**Effort estimate:** 5-8 days.

## 10. Risk Mitigation

### Risk: Compiler bugs produce incorrect SQL

**Mitigation:** The compiled SQL is validated through the same SAVEPOINT execution pipeline as
hand-written SQL. If the generated SQL violates any FK constraint, CHECK constraint, or UNIQUE
index, SQLite rejects it. The compiler never bypasses the existing validation.

### Risk: YAML parsing edge cases (regex patterns with special chars)

**Mitigation:** All regex patterns are stored in single-quoted YAML strings (no escape sequences
except `''` for literal single quote). The compiler includes a roundtrip test: parse YAML, serialize
back to YAML, parse again, and verify identical output.

### Risk: Entity YAML files drift from compiled SQL state

**Mitigation:** A CI check compiles `entities/` to SQL, builds the cache, serializes all entities
back to YAML, and compares with the source files. Any drift is a CI failure.

### Risk: Migration breaks existing praxrr-db consumers

**Mitigation:** The `ops/0.rosettarr.sql` file is regenerated from the compiled entities and
remains in the repository. Consumers that only read SQL ops files see no change. The `entities/`
directory is additive.

### Risk: Incremental ops generation produces incorrect value guards

**Mitigation:** Generated incremental ops are committed as PR drafts requiring human review. The
system never auto-merges generated ops. Complex migrations (multi-entity rename chains, quality
profile regroups) continue to be hand-written in SQL.

**`entities/regular-expressions/DTS-HD-MA.yaml`:**

```yaml

description: ''
regex101Id: null
tags:
  - Audio
  - Lossless Audio
```

**`entities/custom-formats/DTS-HD-MA.yaml`:**

```yaml
name: 'DTS-HD MA'
description: "Matches 'DTS-HD MA' Regex Pattern and negates any other audio types that might conflict."
includeInRename: false
tags:
  - Audio
  - Lossless Audio

conditions:
  - name: 'DTS-HD MA'
    type: release_title
    arrType: all
    negate: false
    required: true
    patterns:
      - name: 'DTS-HD MA'

  - name: 'Not DTS-X'
    type: release_title
    arrType: all
    negate: true
    required: true
    patterns:
      - name: 'DTS-X'

tests: []
```

The quality profile scores for this CF live in the quality profile YAML files
(`entities/quality-profiles/1080p-Remux.yaml`, etc.) under their `customFormatScores` arrays.
This is deliberate: scores belong to the profile-CF relationship, not to the CF entity.

## Appendix B: Rejected Alternatives

### B.1 One mega-file per entity type

Rejected because: Git diffs become unusable when a single 5000-line file changes. Individual entity
files produce clean, reviewable diffs.

### B.2 Scores stored in the CF file instead of the QP file

Rejected because: Scores are a property of the (profile, CF, arr_type) relationship. Storing them
in the CF file would require listing every profile that scores the CF, which creates a maintenance
burden when profiles are added/removed. The QP file already lists its scores, matching the
relational schema (`quality_profile_custom_formats`).

### B.3 Full desired-state model (no incremental SQL ops)

Rejected because: This would replace the append-only event-sourced architecture with a
snapshot-based model. Value guards, conflict detection, and the layered compilation model would all
need to be redesigned. The research unanimously agrees this is architecturally incompatible
(Convergence 8).

### B.4 JSON instead of YAML for authoring

Not rejected, but deprioritized. JSON is valid YAML, so the compiler accepts both. YAML is
preferred for authoring because it supports comments, multi-line strings (for descriptions and regex
patterns), and is more concise. JSON is preferred for machine consumption (API distribution,
TRaSH import).

### B.5 Embedding regex patterns inline in CF conditions

Rejected because: Praxrr's shared regex model is an architectural advantage. Multiple CFs can
reference the same regex by name. Inlining patterns would duplicate regexes and lose the ability
to rename a regex in one place with ON UPDATE CASCADE propagation.
